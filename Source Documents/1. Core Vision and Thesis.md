# 01 — Core Vision and Thesis

## Purpose of this document

This document defines the core vision of the **Bounded Predictive Machine** project after the v0.1/v0.2 learning spikes.

It explains what the system is, what it is not, why the project is restarting from a corrected foundation, and what thesis should guide all future documentation, implementation, and Codex prompts.

This document should be read after:

```text
00-project-continuity-brief.md
```

and before any new implementation work begins.

---

## Project name

**Bounded Predictive Machine**

Short form:

```text
BPM
```

Technical framing:

```text
Bounded Predictive Cognitive Runtime
```

More formal framing:

```text
A world-model-first, signal-source-agnostic, safety-governed predictive cognitive runtime architecture.
```

---

## One-sentence thesis

The Bounded Predictive Machine is a proposed machine cognition architecture in which a system begins from body state, priors, and a world model; generates predictions; observes signals from bounded sources; compares those signals against expectation; updates or abstains based on prediction error and evidence quality; and acts only within explicit safety boundaries.

---

## The central idea

The project is based on a simple but important claim:

```text
A machine should not begin cognition from user text.
A machine should begin from a model of itself and its world.
```

The system should not simply wait for a command and then respond. It should maintain a structured model of:

```text
- what it is
- what state it is in
- what it can sense
- what it is allowed to access
- what it is allowed to change
- what it currently believes
- what it expects to happen next
- what signals would confirm or challenge those expectations
- what remains uncertain
```

From that model, it should generate predictions before treating new input as meaningful.

Then, when a signal arrives, the system should compare the signal against the prediction and ask:

```text
What did I expect?
What happened?
Did the signal match the expectation?
Did it contradict the expectation?
Was the signal reliable enough to matter?
Should the world model change?
Should I abstain?
Should I gather more evidence?
Is any action justified?
```

This is the foundation of the project.

---

## The corrected loop

The corrected architecture starts here:

```text
body/system state + priors + world model
→ prediction
→ multi-source signal intake
→ prediction comparison
→ prediction error
→ evidence quality
→ world-model / belief update or no-update
→ action / no-action
→ safety
→ outcome
→ memory
→ revised model
```

This is the loop that future work must preserve.

The old prototype loop was useful, but it was not the correct foundation:

```text
manual text input
→ prediction
→ signal
→ interpretation
→ evidence
→ belief update / no-update
→ action / no-action
→ safety
→ outcome
→ memory
```

That loop made manual text too central.

The corrected loop makes **model-based prediction** central.

---

## What the system is

The Bounded Predictive Machine is intended to be:

```text
- a bounded runtime
- a predictive system
- a signal-processing architecture
- a world-model-updating system
- a traceable memory system
- an uncertainty-aware system
- a safety-governed action system
- a pre-agent cognitive substrate
```

It is designed to explore whether cognition-like behavior can be built from disciplined runtime structure rather than from a large language model acting as the central mind.

The project does not assume that intelligence begins with language.

It assumes that a machine needs a disciplined relationship between:

```text
model
prediction
signal
error
update
memory
action
safety
```

---

## What the system is not

The system is not intended to be:

```text
- a chatbot
- an LLM wrapper
- a prompt chain
- a normal AI agent
- a tool-use loop
- a memory database
- a transformer replacement
- a consciousness model
- a personality system
- an uncontrolled autonomous assistant
```

It may eventually contain language models, tools, memory systems, sensors, and action modules.

But those components must be admitted into the architecture under explicit boundaries.

They are not the architecture itself.

The runtime owns the loop.

A model may assist a stage.

---

## Why an LLM is not the mind of the system

A large language model may be useful later, but it must not become the core cognitive authority.

An LLM may eventually help with:

```text
- proposing interpretations
- summarizing traces
- generating explanations
- detecting ambiguity
- translating records into readable language
- suggesting possible predictions
- helping a human inspect system state
```

But an LLM should not be allowed to own:

```text
- the world model
- belief authority
- memory truth
- safety permission
- action authorization
- boundary definition
- capability admission
- causal certainty
```

The system should not trust a model output simply because it sounds coherent.

A model output is a signal or proposal.

It is not truth.

It must be bounded, checked, traced, and compared against the system’s existing model and evidence rules.

---

## Signals are not truth

A signal is anything the system receives or observes.

Possible signal sources include:

```text
- manual text input
- camera input
- microphone input
- operating system state
- active window or application state
- file changes
- terminal output
- process state
- resource load
- memory traces
- internal runtime state
- network state, if explicitly allowed later
```

The system should not treat a signal as automatically true.

It should ask:

```text
Where did this signal come from?
Is this source allowed?
Is this source reliable?
Is the signal fresh?
Is the signal ambiguous?
Does it match what was predicted?
Does it contradict the current model?
Is the evidence strong enough to update anything?
```

Manual text is only one signal source.

It is not the root of cognition.

---

## Body state matters

The machine must have a model of its own body.

For this project, body does not mean a human body. It means the operational state of the machine/runtime.

Body state may include:

```text
- runtime identity
- current process state
- project directory
- available files
- allowed paths
- available signal sources
- enabled capabilities
- safety boundaries
- current resource state
- current memory state
- current uncertainty state
```

The system must understand what it is allowed to observe and what it is allowed to affect.

A machine that does not know its own boundary should not be given broad autonomy.

---

## Priors matter

The system also needs priors.

A prior is a structural assumption that exists before a new signal arrives.

Examples:

```text
- signals may be noisy
- manual text may be incomplete
- memory retrieval is context, not truth
- unsafe action should be blocked
- uncertainty should prevent overconfident updates
- external sensors require explicit permission
- terminal access should begin as observation-only
- source documents should remain read-only unless explicitly allowed
```

Priors are not permanent truths.

They are starting assumptions that shape prediction and can later be revised if evidence justifies it.

---

## The world model matters

The world model is the system’s current structured understanding of its environment, task, user context, project state, and constraints.

It should include things like:

```text
- current project phase
- known active goals
- known constraints
- available signal sources
- expected near-future events
- relevant memory traces
- current confidence levels
- unresolved uncertainty
```

Predictions should be generated from this world model.

The system should not do this:

```text
input says X
→ predict X
→ observe X
```

That is circular.

The system should do this:

```text
current model expects X
→ signal shows Y
→ compare X and Y
→ produce prediction error
→ update, abstain, or gather more evidence
```

That is the corrected architecture.

---

## Prediction is the heart of the system

A prediction is not a guess copied from the latest user message.

A prediction is a structured expectation generated before the next signal is interpreted.

A prediction may describe:

```text
- expected signal source
- expected signal type
- expected signal features
- expected system state change
- expected user behavior
- expected file state
- expected runtime condition
- expected uncertainty
- expected outcome of an action
```

Prediction gives the system something to test.

Without prediction, the system can only react.

With prediction, the system can compare expectation with observation and improve its model.

---

## Prediction error drives learning

Prediction error is the difference between what the system expected and what it observed.

A signal may produce:

```text
- match
- mismatch
- partial match
- inconclusive result
- contradiction
- missing evidence
- unexpected source
- unexpected timing
```

Prediction error does not automatically mean the world model should change.

The system still needs to judge evidence quality.

A weak or ambiguous signal may produce no update.

A strong signal may justify model revision.

A contradictory signal may require the system to gather more evidence before changing anything.

---

## Evidence quality matters

The system should not update simply because something unexpected happened.

It should evaluate the quality of the evidence.

Evidence quality may depend on:

```text
- source reliability
- signal clarity
- signal freshness
- match or mismatch strength
- uncertainty count
- consistency with other records
- whether the signal is admissible
- whether the observation can be repeated or verified
```

The system should be able to say:

```text
This signal arrived.
This is where it came from.
This is what I predicted.
This is how the signal compared to the prediction.
This is how reliable the evidence was.
This is why I updated or did not update.
```

That traceability is part of the thesis.

---

## No-update is intelligent

A mature system should not update its model every time a signal appears.

Sometimes the best outcome is:

```text
NoUpdate
```

No-update is appropriate when:

```text
- evidence is weak
- signal meaning is unclear
- source reliability is low
- the signal is outside the boundary
- prediction comparison is inconclusive
- the update target is not clear
- memory is relevant but not decisive
```

The system should preserve the signal and uncertainty without corrupting the world model.

---

## No-action is intelligent

A mature system should not act merely because action is possible.

Sometimes the best outcome is:

```text
NoAction
```

No-action is appropriate when:

```text
- permission is missing
- safety boundary is unclear
- action effect cannot be predicted
- evidence is too weak
- the system is uncertain
- the action target is ambiguous
- the system cannot observe the outcome
- human approval is required
```

The system may observe before it acts.

The system may propose before it executes.

The system may abstain before it risks doing harm.

---

## Memory is reconstruction, not storage

Memory should not be treated as simple storage.

A memory trace should help reconstruct what happened.

A good memory record should preserve links between:

```text
- prior model state
- prediction
- signal
- interpretation
- prediction error
- evidence quality
- update or no-update
- action or no-action
- safety check
- outcome
- remaining uncertainty
```

Memory retrieval should not automatically become truth.

Retrieved memory is context.

It may inform prediction, but it should still be checked against current signals and current boundaries.

---

## Safety is structural, not decorative

Safety is not a wrapper added after the cognitive loop.

Safety is part of the architecture.

The system must track:

```text
- what it can observe
- what it can change
- what is read-only
- what requires permission
- what requires human approval
- what can be recovered
- what must be logged
- what capabilities are admitted
- what capabilities are not admitted
```

The safety shell should govern:

```text
- sensor access
- file access
- terminal access
- external actions
- memory writes
- model updates
- capability expansion
```

A capability should not be available merely because the machine technically can perform it.

It must be admitted, scoped, traceable, and revocable.

---

## Why the project restarted

The v0.1/v0.2 prototypes were useful, but they encoded the wrong foundation.

They proved valuable mechanics:

```text
- records
- serialization
- JSONL traces
- schema validation
- tests
- abstention
- no-update
- no-action
- uncertainty gates
- evidence scoring
- local memory retrieval
- CLI observability
```

But they centered the loop on manual text input.

That made the prototype a structured manual-text processing system rather than a true world-model-first predictive runtime.

The restart exists to correct that foundation.

The new foundation must make this true from the first line:

```text
body state + priors + world model
→ prediction
→ signal
→ prediction error
```

---

## The long-term ambition

The long-term ambition is to explore whether a machine can become more cognitively disciplined by maintaining a bounded predictive loop.

The aim is not to make the system seem intelligent as quickly as possible.

The aim is to make it structurally careful.

The system should become better at:

```text
- knowing what it is
- knowing what it can sense
- knowing what it expected
- noticing mismatch
- evaluating evidence
- updating carefully
- refusing weak updates
- refusing unsafe actions
- remembering why things changed
- using memory without overtrusting it
- monitoring its own uncertainty
- expanding capability only under governance
```

The system should become more useful only as its boundaries, predictions, evidence handling, memory, and safety become more reliable.

---

## Design principles

Future documentation and implementation should preserve these principles.

### 1. Model first

The system starts from body state, priors, and world model.

It does not start from manual text.

### 2. Prediction before signal interpretation

Signals should be compared against predictions generated before interpretation.

### 3. Signals are not truth

Signals are observations or inputs. They require source, boundary, reliability, and evidence checks.

### 4. Manual text is one signal source

Manual text is useful for testing, but it is not the architecture’s center.

### 5. Prediction error drives revision

Model change should be caused by comparison between expectation and observation, not by input alone.

### 6. Evidence quality gates updates

Prediction error creates update pressure, but evidence quality decides whether update is justified.

### 7. No-update and no-action are valid outcomes

A system that refuses to update or act under uncertainty is behaving intelligently.

### 8. Memory is reconstructive

Memory should preserve the evidence chain, not just summaries.

### 9. Safety governs capability

A capability must be admitted, bounded, traceable, and revocable before use.

### 10. LLMs are optional components

Language models may later assist bounded stages, but they are not the system’s mind.

---

## What future work must not do

Future work should not drift back into these mistakes:

```text
- treating manual text as the root of cognition
- deriving prediction from the incoming input
- treating interpretation as the main cognitive act
- adding memory retrieval before fixing the model-first loop
- adding embeddings before the prediction-error architecture is correct
- adding camera/mic/window access before permission and boundaries exist
- adding autonomous terminal actions before safety design exists
- using an LLM as belief owner or action authority
- optimizing CLI polish before the loop shape is correct
```

The first stages matter.

If the first stages are wrong, later intelligence is built on the wrong shape.

---

## What the corrected first prototype should prove

The corrected first prototype should prove one thing:

```text
Can the system generate a prediction from body state, priors, and world model before receiving or interpreting a signal, then compare the signal against that prediction and produce update/no-update behavior?
```

It does not need to be impressive.

It does not need broad sensors.

It does not need embeddings.

It does not need an LLM.

It does not need autonomous tools.

It only needs to prove the corrected foundation.

A minimal successful loop would show:

```text
BodyStateRecord
PriorModelRecord
WorldModelRecord
PredictionRecord
SignalSourceRecord
SignalRecord
PredictionErrorRecord
EvidenceRecord
ModelUpdateRecord or NoUpdateRecord
ActionProposalRecord or NoActionRecord
SafetyCheckRecord
OutcomeRecord
MemoryTraceRecord
LoopRecord
```

The order matters.

The prediction must come before the signal is interpreted.

The signal must be compared against the prediction.

The update must be justified by evidence.

---

## Final statement

The Bounded Predictive Machine project is an attempt to build machine cognition from disciplined first principles.

It asks:

```text
What must a machine know about itself and its world before it should update, remember, or act?
```

The answer is not:

```text
Give it a language model and tools.
```

The answer is:

```text
Give it a bounded body model, priors, a world model, predictions, signal discipline, prediction error, evidence checks, uncertainty tracking, traceable memory, and safety-governed action.
```

That is the core vision.

That is the thesis.
