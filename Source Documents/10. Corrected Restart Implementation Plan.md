# 10 — Corrected Restart Implementation Plan

## Purpose of this document

This document defines the implementation plan for restarting the **Bounded Predictive Machine** prototype from the corrected foundation.

The key instruction is:

```text
Do not continue the v0.2 manual-text-first architecture as the foundation.
```

v0.1 and v0.2 were useful learning spikes.

They proved important mechanics, but they also revealed that the first loop shape was wrong for the deeper project goal.

The corrected restart must begin with:

```text
body/system state + priors + world model
→ prediction
→ signal intake
→ prediction comparison
→ prediction error
→ evidence quality
→ model update or no-update
→ action or no-action
→ safety
→ outcome
→ memory
→ revised model
```

This document turns that corrected architecture into a staged implementation plan.

## Why this is a restart, not v0.3

This next implementation should not be called v0.3.

v0.3 would imply continuation of the v0.2 foundation.

But the project decision is different:

```text
Restart from corrected foundation.
```

Recommended branch name:

```text
restart/v0.1-world-model-first
```

This communicates that the new work is a corrected first foundation, not the next layer on top of the old manual-text-first loop.

## Main implementation principle

The first implementation must prove the corrected order.

The first priority is not intelligence, polish, memory retrieval, embeddings, sensors, or tool use.

The first priority is this order:

```text
BodyStateRecord
PriorModelRecord
WorldModelRecord
→ PredictionRecord
→ SignalSourceRecord
→ SignalRecord
→ PredictionErrorRecord
→ EvidenceRecord
→ ModelUpdateRecord or NoUpdateRecord
→ ActionProposalRecord or NoActionRecord
→ SafetyCheckRecord
→ OutcomeRecord
→ MemoryTraceRecord
→ LoopRecord
```

If this order is wrong, every later feature inherits the wrong foundation.

## Common implementation risks

### Risk 1: Reusing old manual loop too directly

Do not wrap the old manual loop and call it corrected.

The new loop must begin with body/prior/world records.

### Risk 2: Prediction after input

If prediction is generated after manual text arrives, the architecture is wrong.

### Risk 3: Signal source abstraction too late

Manual text must become one signal source early.

### Risk 4: Memory too early

Do not add retrieval before prediction error is correct.

### Risk 5: Schema validation too early

Do not freeze unstable records with schemas before the corrected structure settles.

### Risk 6: CLI drives architecture

CLI should expose the loop, not define it.

### Risk 7: Safety added too late

Safety boundaries should be represented from the first body state.
