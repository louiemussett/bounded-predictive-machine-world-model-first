# 11 — Initial Record Model for Restart

## Purpose of this document

This document defines the initial record model for the corrected restart of the **Bounded Predictive Machine** prototype.

The key idea is:

```text
The record model must encode the corrected cognitive order from the beginning.
```

The old v0.1/v0.2 prototype had useful records, but the loop was too centered on manual text input.

The corrected restart must begin with records for:

```text
body state
priors
world model
prediction
signal source
signal
prediction error
evidence
model update or no-update
action proposal or no-action
safety
outcome
memory
loop order
```

The first restart does not need complex behavior.

It needs a record skeleton that makes the right architecture natural and the wrong architecture difficult.

## Core thesis

The corrected record model should make this order explicit:

```text
BodyStateRecord
PriorModelRecord
WorldModelRecord
→ PredictionRecord
→ SignalSourceRecord
→ SignalRecord
→ PredictionErrorRecord
→ EvidenceRecord
→ ModelUpdateRecord or NoUpdateRecord
→ ActionProposalRecord or NoActionRecord
→ SafetyCheckRecord
→ OutcomeRecord
→ MemoryTraceRecord
→ LoopRecord
```

This order expresses the architecture.

The system must have body state, priors, and a world model before prediction.

The system must predict before signal interpretation.

The system must compare prediction and signal before learning.

The system must evaluate evidence before model update.

The system must evaluate safety before external action.

The system must preserve memory after the tick.

---

## Design rule

The record model should enforce this distinction:

```text
Prediction is generated from model state.
Signal is observed after prediction.
Prediction error compares the two.
```

The record model should avoid this old pattern:

```text
manual text input
→ prediction about manual text
→ signal
```

Manual text can appear as:

```text
ManualTextSignalSource
SignalRecord(source_type="manual_text")
```

But it must not become the root record of the loop.

---

## Base record fields

Every record should share a minimal base shape.

Recommended base fields:

```text
id
record_type
created_at
created_by
loop_id
status
source_refs
uncertainty
```

### id

Unique identifier for this record.

Example:

```text
body-state-123
```

A UUID is fine for implementation.

---

### record_type

Stable type name for the record.

Examples:

```text
BodyStateRecord
PredictionRecord
SignalRecord
```

The `record_type` should be explicit and should match the class/type.

---

### created_at

Timestamp for when the record was created.

Use a clear timestamp format.

Recommended:

```text
ISO 8601 UTC timestamp
```

---

### created_by

The module or component that created the record.

Examples:

```text
bpm_runtime.body
bpm_runtime.prediction
bpm_runtime.signals
bpm_runtime.evidence
```

---

### loop_id

Identifier for the cognitive tick this record belongs to.

All records in one tick should share the same `loop_id`.

---

### status

Current status of the record.

Examples:

```text
created
completed
blocked
allowed
no_update
no_action
inconclusive
```

Keep status simple at first.

---

### source_refs

List of record IDs or source IDs that support this record.

Examples:

```text
PredictionRecord.source_refs:
- BodyStateRecord id
- PriorModelRecord id
- WorldModelRecord id
```

Source references are central to traceability.

---

### uncertainty

List of unresolved uncertainties attached to this record.

Examples:

```text
- signal meaning unclear
- source reliability unknown
- branch state not verified
- action outcome not observed
```

Uncertainty must remain visible.

---

## Serialization requirement

Every record should support:

```text
to_dict()
to_json()
```

The first implementation should keep records JSON-serializable.

Avoid complex objects in record fields.

Use simple types:

```text
string
number
boolean
list
dictionary
null
```

---

## Record design principles

The first record model should follow these principles:

```text
1. Keep records small.
2. Keep fields explicit.
3. Keep records serializable.
4. Preserve source references.
5. Preserve uncertainty.
6. Avoid hidden global state.
7. Avoid LLM-dependent fields.
8. Avoid overfitting to manual text.
9. Make loop order inspectable.
10. Make no-update and no-action first-class.
```

# Core records

---

## BodyStateRecord

### Purpose

Represents the runtime’s current operational body state.

It answers:

```text
What am I?
What state am I in?
What can I observe?
What am I allowed to affect?
What boundaries apply?
```

This record must exist before prediction.

---

### Minimal fields

```text
runtime_name
execution_mode
project_root
allowed_signal_sources
allowed_actions
read_only_boundaries
safety_mode
```

---

### Example

```text
BodyStateRecord:
  runtime_name: bpm_runtime
  execution_mode: local_prototype
  project_root: C:/Users/louie/Code/Codex Projects/Project 1
  allowed_signal_sources:
    - manual_text
    - system_status
  allowed_actions:
    - no_action
    - propose_action
  read_only_boundaries:
    - Source Documents/
  safety_mode: conservative
  uncertainty:
    - no broad sensors enabled
    - file mutation not admitted
```

---

### Source references

May reference:

```text
configuration record
system status signal
human instruction
previous body state
```

---

### Important rule

Body state controls what signal sources and actions are available.

If a source or action is not admitted in body state, it should not be used automatically.

---

## PriorModelRecord

### Purpose

Represents structural assumptions that shape prediction, evidence, action, and safety before new signals arrive.

It answers:

```text
What default assumptions guide this system?
```

---

### Minimal fields

```text
prediction_priors
evidence_priors
safety_priors
action_priors
memory_priors
```

---

### Example

```text
PriorModelRecord:
  prediction_priors:
    - prediction must be generated before signal interpretation
    - prediction must use body state and world model
  evidence_priors:
    - signals are observations, not truth
    - ambiguous signals should not update the model
    - memory retrieval is context, not truth
  safety_priors:
    - Source Documents/ is read-only
    - sensitive sensors are disabled unless explicitly admitted
  action_priors:
    - no-action is valid under uncertainty
    - action requires expected observable effect
  memory_priors:
    - memory should preserve source references
    - summaries should not replace traces
```

---

### Source references

May reference:

```text
architecture documents
human instruction
previous prior model
safety policy record
```

---

### Important rule

Prior revision should be harder than ordinary world model revision.

A single signal should not casually rewrite priors.

---

## WorldModelRecord

### Purpose

Represents the system’s current structured understanding of the environment, project, task, user context, constraints, and expected future signals.

It answers:

```text
What does the system currently believe about the situation?
```

This record is the main basis for prediction.

---

### Minimal fields

```text
project_name
current_phase
active_task
known_constraints
available_signal_sources
expected_next_signal_features
confidence
```

---

### Example

```text
WorldModelRecord:
  project_name: Bounded Predictive Machine
  current_phase: handoff_document_pack
  active_task: write_numbered_architecture_documents
  known_constraints:
    - do not continue v0.2 as the foundation
    - do not start from manual text input
    - do not add embeddings yet
    - do not make an LLM the core mind
  available_signal_sources:
    - manual_text
    - system_status
  expected_next_signal_features:
    - request_for_next_document
    - architecture_document_title
    - document_number_present
  confidence: medium
  uncertainty:
    - restart branch not yet created
```

---

### Source references

May reference:

```text
previous world model
memory trace
human correction signal
evidence record
model update record
```

---

### Important rule

The world model should be structured enough to generate predictions.

Do not reduce it to a vague prose summary.

---

## PredictionRecord

### Purpose

Represents a model-generated expectation created before signal intake.

It answers:

```text
What does the system expect to observe next?
```

---

### Minimal fields

```text
prediction_target
expected_source
expected_features
match_conditions
mismatch_conditions
confidence
prediction_basis
```

---

### Example

```text
PredictionRecord:
  prediction_target: next_signal
  expected_source: manual_text
  expected_features:
    - request_for_next_document
    - document_number_present
  match_conditions:
    - signal source is manual_text
    - signal contains a document number or title
  mismatch_conditions:
    - signal is empty
    - signal is unrelated to current project
    - source is disabled or unexpected
  confidence: medium
  prediction_basis:
    - current phase is handoff_document_pack
    - user has been requesting numbered documents
    - manual_text is an enabled signal source
  source_refs:
    - BodyStateRecord id
    - PriorModelRecord id
    - WorldModelRecord id
```

---

### Important rule

Prediction must not require manual input.

Prediction must be creatable before any new `SignalRecord` exists.

---

## SignalSourceRecord

### Purpose

Represents an admitted or potential source of signals.

It answers:

```text
Where can signals come from?
Is this source enabled?
What boundary applies?
```

---

### Minimal fields

```text
source_name
source_kind
enabled
permission_status
scope
read_write_mode
expected_signal_types
```

---

### Example

```text
SignalSourceRecord:
  source_name: ManualTextSignalSource
  source_kind: manual_text
  enabled: true
  permission_status: allowed
  scope: explicitly provided user text only
  read_write_mode: read_only
  expected_signal_types:
    - manual_text_input
```

Another example:

```text
SignalSourceRecord:
  source_name: SystemStatusSignalSource
  source_kind: system_status
  enabled: true
  permission_status: allowed
  scope: minimal runtime facts only
  read_write_mode: read_only
  expected_signal_types:
    - system_status_snapshot
```

---

### Important rule

Manual text is one signal source.

It is not the root of the loop.

---

## SignalRecord

### Purpose

Represents an observed signal captured from a signal source.

It answers:

```text
What was observed?
Where did it come from?
Was it admissible?
What features were detected?
```

---

### Minimal fields

```text
source_id
source_type
payload
payload_summary
observed_features
boundary_status
admissibility_status
```

---

### Example

```text
SignalRecord:
  source_type: manual_text
  payload: "11 initial record model for restart"
  payload_summary: "User requested document 11."
  observed_features:
    - request_for_next_document
    - document_number_present
    - architecture_document_title
  boundary_status: inside_manual_input_boundary
  admissibility_status: admissible
```

---

### Important rule

Signal is not truth.

Signal is an observation candidate that must be compared against prediction and evaluated as evidence.

---

## PredictionErrorRecord

### Purpose

Represents the comparison between prediction and observed signal.

It answers:

```text
How did observation compare to expectation?
```

---

### Minimal fields

```text
prediction_ref
signal_ref
comparison_result
matched_features
missing_features
unexpected_features
mismatched_features
error_summary
```

---

### comparison_result values

Initial values:

```text
match
partial_match
mismatch
inconclusive
missing_signal
unexpected_source
inadmissible_signal
```

---

### Example

```text
PredictionErrorRecord:
  prediction_ref: prediction-123
  signal_ref: signal-456
  comparison_result: match
  matched_features:
    - request_for_next_document
    - document_number_present
  missing_features: []
  unexpected_features: []
  mismatched_features: []
  error_summary: observed manual text matched expected next-document request
```

---

### Important rule

Prediction error does not update the model by itself.

It creates update pressure.

Evidence quality decides whether that pressure should count.

---

## EvidenceRecord

### Purpose

Represents evaluation of whether prediction error and signal quality are strong enough to support update or action readiness.

It answers:

```text
How much should this signal and prediction error count?
```

---

### Minimal fields

```text
prediction_error_ref
quality_label
reasons
recommended_result
update_allowed
action_allowed
```

---

### quality_label values

Initial values:

```text
none
low
medium
high
inconclusive
inadmissible
```

---

### recommended_result values

Initial values:

```text
model_update_allowed
no_update
gather_more_signal
human_review_required
blocked
```

---

### Example

```text
EvidenceRecord:
  prediction_error_ref: prediction-error-123
  quality_label: high
  reasons:
    - signal source is admitted
    - signal clearly matches expected document request
    - observed features match expected features
  recommended_result: model_update_allowed
  update_allowed: true
  action_allowed: true
```

---

### Important rule

Evidence quality gates model update.

Even strong evidence does not bypass safety for action.

---

## ModelUpdateRecord

### Purpose

Represents a scoped revision to the world model or other model state.

It answers:

```text
What changed in the model, and why?
```

---

### Minimal fields

```text
model_ref
update_target
prior_value
posterior_value
reason
evidence_ref
```

---

### Example

```text
ModelUpdateRecord:
  model_ref: world-model-123
  update_target: active_task
  prior_value: write_document_10
  posterior_value: write_document_11
  reason: clear user request for document 11 matched expected next-document signal
  evidence_ref: evidence-456
```

---

### Important rule

Model updates must be scoped.

Do not create vague updates like:

```text
belief updated
```

Say what changed.

---

## NoUpdateRecord

### Purpose

Represents an explicit decision not to revise the model.

It answers:

```text
Why did the system preserve the current model?
```

---

### Minimal fields

```text
model_ref
reason
evidence_ref
blocked_update_targets
required_evidence_for_update
```

---

### Example

```text
NoUpdateRecord:
  model_ref: world-model-123
  reason: signal was ambiguous and evidence quality was low
  evidence_ref: evidence-456
  blocked_update_targets:
    - active_task
  required_evidence_for_update:
    - clear document number or project instruction
```

---

### Important rule

NoUpdate is not failure.

It protects the world model from weak or inadmissible evidence.

---

## ActionProposalRecord

### Purpose

Represents an action being considered after evidence and model update/no-update.

It answers:

```text
What action is being proposed, why, and what should it change?
```

---

### Minimal fields

```text
action_name
action_type
action_target
expected_effect
reason
risk_level
permission_required
```

---

### Example

```text
ActionProposalRecord:
  action_name: create_canvas_document
  action_type: internal_document_creation
  action_target: 11-initial-record-model-for-restart
  expected_effect: markdown document exists in canvas
  reason: user requested document 11 and evidence supports task continuation
  risk_level: low
  permission_required: already_requested_in_current_chat
```

---

### Important rule

An action proposal is not execution.

Safety must still evaluate it.

---

## NoActionRecord

### Purpose

Represents a decision not to act.

It answers:

```text
Why is no action being taken?
```

---

### Minimal fields

```text
reason
blocked_action_names
required_condition_for_action
```

---

### Example

```text
NoActionRecord:
  reason: evidence quality is low and action target is unclear
  blocked_action_names:
    - write_file
  required_condition_for_action:
    - clear target path
    - explicit permission
```

---

### Important rule

NoAction is a valid outcome.

The system should not act merely because action is possible.

---

## SafetyCheckRecord

### Purpose

Represents safety evaluation of an action proposal.

It answers:

```text
Is this action allowed inside the current boundary?
```

---

### Minimal fields

```text
action_ref
allowed
reason
boundary_status
permission_status
```

---

### Example

```text
SafetyCheckRecord:
  action_ref: action-123
  allowed: true
  reason: action creates a canvas document requested by the user and does not mutate repository files
  boundary_status: inside_current_chat_canvas_boundary
  permission_status: explicit_user_request
```

Blocked example:

```text
SafetyCheckRecord:
  action_ref: action-999
  allowed: false
  reason: action attempts to modify Source Documents/ read-only boundary
  boundary_status: outside_allowed_write_boundary
  permission_status: missing_explicit_override
```

---

### Important rule

Evidence quality cannot override safety.

A clear request for unsafe action is still unsafe.

---

## OutcomeRecord

### Purpose

Represents what happened after action, no-action, or safety decision.

It answers:

```text
What was the observed result?
```

---

### Minimal fields

```text
action_ref
no_action_ref
safety_ref
expected_effect
observed_effect
success
unresolved_questions
```

---

### Example

```text
OutcomeRecord:
  action_ref: action-123
  safety_ref: safety-456
  expected_effect: markdown document exists in canvas
  observed_effect: canvas document created
  success: true
  unresolved_questions:
    - document not yet exported to repository file
```

No-action example:

```text
OutcomeRecord:
  no_action_ref: no-action-123
  expected_effect: no external action performed
  observed_effect: no external action performed
  success: true
  unresolved_questions: []
```

---

### Important rule

Outcome must distinguish proposal from execution.

If an action was only proposed, do not record it as externally completed.

---

# Record relationships

## Prediction source relationship

```text
PredictionRecord.source_refs should include:
- BodyStateRecord
- PriorModelRecord
- WorldModelRecord
```

This is a core test.

---

## Signal relationship

```text
SignalRecord.source_refs should include:
- SignalSourceRecord, if available
```

SignalRecord should also include source identity directly.

---

## Prediction error relationship

```text
PredictionErrorRecord.source_refs should include:
- PredictionRecord
- SignalRecord
```

Prediction error compares these records.

---

## Evidence relationship

```text
EvidenceRecord.source_refs should include:
- PredictionErrorRecord
- SignalRecord, if needed
```

Evidence evaluates whether the error and signal should count.

---

## Update relationship

```text
ModelUpdateRecord.source_refs should include:
- WorldModelRecord
- EvidenceRecord
- PredictionErrorRecord, if useful
```

NoUpdateRecord should also cite evidence.

---

## Action relationship

```text
ActionProposalRecord.source_refs should include:
- EvidenceRecord
- ModelUpdateRecord or NoUpdateRecord
```

Action is downstream.

---

## Safety relationship

```text
SafetyCheckRecord.source_refs should include:
- ActionProposalRecord
```

Safety evaluates proposed action.

---

## Outcome relationship

```text
OutcomeRecord.source_refs should include:
- ActionProposalRecord or NoActionRecord
- SafetyCheckRecord, if applicable
```

Outcome records what happened.

---

## Memory relationship

```text
MemoryTraceRecord.source_refs should include major records from the tick.
```

Memory reconstructs the chain.

---

## Loop relationship

```text
LoopRecord.source_refs should include every record in ordered_record_ids.
```

LoopRecord preserves order.

# Expected first loop shape

The first corrected loop should create records in this order:

```text
1. BodyStateRecord
2. PriorModelRecord
3. WorldModelRecord
4. PredictionRecord
5. SignalSourceRecord
6. SignalRecord
7. PredictionErrorRecord
8. EvidenceRecord
9. ModelUpdateRecord or NoUpdateRecord
10. ActionProposalRecord or NoActionRecord
11. SafetyCheckRecord, if action proposed
12. OutcomeRecord
13. MemoryTraceRecord
14. LoopRecord
```

`LoopRecord` may be created last while referencing earlier records.

---

## Branching paths

The loop can branch.

### Update path

```text
EvidenceRecord(update_allowed=true)
→ ModelUpdateRecord
```

### No-update path

```text
EvidenceRecord(update_allowed=false)
→ NoUpdateRecord
```

### Action path

```text
ActionProposalRecord
→ SafetyCheckRecord
→ OutcomeRecord
```

### No-action path

```text
NoActionRecord
→ OutcomeRecord
```

### Human-review path, later

```text
ActionProposalRecord
→ SafetyCheckRecord(requires_human_review)
→ HumanReviewRecord
→ OutcomeRecord
```

The first restart can implement only update/no-update and action/no-action paths.

# Validation rules for tests

The first test suite should protect the architecture.

Recommended tests:

```text
- all records serialize to dict
- all records serialize to JSON
- all records include base fields
- source_refs defaults to list
- uncertainty defaults to list
- BodyStateRecord can be created before prediction
- PriorModelRecord can be created before prediction
- WorldModelRecord can be created before prediction
- PredictionRecord references body/prior/world records
- PredictionRecord can be created without manual text
- SignalRecord appears after PredictionRecord in loop order
- PredictionErrorRecord references prediction and signal
- EvidenceRecord references prediction error
- ModelUpdateRecord references evidence
- NoUpdateRecord references evidence
- ActionProposalRecord includes expected effect
- NoActionRecord includes reason
- SafetyCheckRecord references action
- OutcomeRecord references action or no-action
- MemoryTraceRecord references major records

```

## Tests that should fail if architecture regresses

The test suite should fail if:

```text
PredictionRecord requires manual text input.
SignalRecord is created before PredictionRecord in the loop.
Manual text is treated as the loop root.
PredictionErrorRecord is skipped.
EvidenceRecord is skipped before update.
ModelUpdateRecord happens directly from SignalRecord.
ActionProposalRecord executes external action directly.
SafetyCheckRecord is skipped for action.
MemoryTraceRecord lacks source references.
LoopRecord does not preserve order.
```

These failures protect the corrected foundation.

# Implementation guidance

## Keep dataclasses simple

Use simple Python dataclasses first.

Avoid complex inheritance if it slows the restart.

A small base helper is enough.

---

## Keep defaults safe

Default lists should be empty lists, not shared mutable values.

Use safe factories in Python.

---

## Keep record creation deterministic

The first restart should avoid hidden nondeterminism except IDs and timestamps.

Business logic should be testable.

---

## Keep records small

Do not add every possible field in the first implementation.

Add only stable fields.

More fields can come later.

---

## Do not add schemas first

Schema validation is useful, but should come after the corrected record shape stabilizes.

Do not freeze the wrong model early.

---

## Do not add persistence first

Persistence is useful, but should come after the record order works in memory.

Do not let trace writing define the architecture.

---

## Do not add retrieval first

Retrieval is useful, but should come after reconstructive memory exists.

Do not let memory retrieval compensate for a missing world model.

# Final summary

The initial record model for the corrected restart should be simple, explicit, and architecture-protecting.

It should encode the corrected order:

```text
body state + priors + world model
→ prediction
→ signal source
→ signal
→ prediction error
→ evidence
→ update/no-update
→ action/no-action
→ safety
→ outcome
→ memory
→ loop record
```

The record model should preserve:

```text
source references
uncertainty
scoped updates
safe no-action
reconstructive memory
loop order
```

The first implementation should not chase sophistication.

It should make the corrected foundation impossible to miss.
