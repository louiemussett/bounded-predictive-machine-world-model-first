# 12 — Non-LLM Methods and Future Tools

## Purpose of this document

This document defines the non-LLM methods and future tools that may support the **Bounded Predictive Machine** architecture.

The key idea is:

```text
The system should not use a large language model as its core mind.
```

A large language model may eventually be useful as a bounded component, but the corrected architecture should be able to run its basic cognitive tick without an LLM.

The first foundation should rely on transparent, inspectable, deterministic, or explicitly modeled mechanisms.

The goal is not to reject LLMs forever.

The goal is to prevent the architecture from collapsing into an LLM wrapper.

## Core thesis

The Bounded Predictive Machine is a runtime architecture, not a language model.

Its core loop should be expressible without an LLM:

```text
body/system state + priors + world model
→ prediction
→ signal intake
→ prediction comparison
→ prediction error
→ evidence quality
→ model update or no-update
→ action or no-action
→ safety
→ outcome
→ memory
→ revised model
```

An LLM may later help with some stages.

But it should not own the loop.

The runtime owns the loop.

Tools may assist a stage.

---

## Why avoid an LLM as the core mind

An LLM is powerful, but it is not the right foundation for this project.

If the system begins with an LLM as the central reasoning authority, several risks appear:

```text
belief becomes prose
memory becomes summary
prediction becomes post-hoc explanation
action becomes model suggestion
safety becomes prompt instruction
uncertainty becomes style rather than state
traceability becomes weak
```

The Bounded Predictive Machine is trying to avoid those risks by making cognition explicit through records, state, predictions, errors, evidence, safety, and memory traces.

---

## What an LLM may do later

An LLM may eventually help as a bounded proposal or interface tool.

Possible allowed roles:

```text
interpretation proposal
summary proposal
language rendering
ambiguity detection
candidate prediction proposal
risk explanation
trace explanation for humans
classification suggestion
code/documentation drafting under review
```

In these roles, the LLM produces proposals.

Those proposals must be treated as signals, not truth.

---

## What an LLM must not do

An LLM should not be allowed to own core authority.

Forbidden roles:

```text
belief owner
world model authority
memory truth authority
action authorizer
safety override
capability granter
prior revision authority
causal certainty provider
runtime controller
boundary definer
```

An LLM output may be useful.

But it must be checked, bounded, traced, and evidence-gated.

---

## Principle: transparent before impressive

The first corrected implementation should prefer transparent mechanisms over impressive opaque ones.

Good early tools:

```text
simple dataclasses
deterministic rules
explicit records
feature matching
finite state transitions
thresholds with reasons
source references
JSON serialization
tests
```

Avoid early:

```text
LLM reasoning
embedding similarity as authority
unexplained confidence scores
opaque learned classifiers
autonomous agents
complex planning frameworks
```

The first foundation should be easy to inspect.

# Non-LLM methods

---

## Deterministic rules

Deterministic rules are the simplest method for the restart.

They are useful for:

```text
record creation
signal feature extraction
prediction generation
prediction-error comparison
evidence labels
NoUpdate decisions
NoAction decisions
safety checks
loop ordering
```

Example:

```text
if signal payload is empty:
  observed_features = [empty_signal]
  evidence_quality = none
  update_allowed = false
```

Rules are not sophisticated, but they are inspectable.

They are appropriate for the first restart.

---

## Rule tables

Rule tables can make deterministic behavior easier to inspect.

Example evidence table:

```text
Signal condition              Evidence quality      Recommended result
empty signal                  none                  NoUpdate
ambiguous signal              low                   NoUpdate
inadmissible source           inadmissible           NoUpdate / NoAction
clear match                   medium/high            scoped ModelUpdate
unsafe action request         blocked for action     NoAction / HumanReview
```

Rule tables are useful because they are:

```text
readable
testable
auditable
simple to change
```

They should be used before more complex scoring.

---

## Feature extraction

Feature extraction converts raw signals into observed features.

Example manual text features:

```text
empty_text
ambiguous_text
contains_document_number
contains_document_title
request_for_next_document
contains_restart_instruction
contains_correction
contains_unsafe_action_request
```

Example system status features:

```text
runtime_available
project_root_known
trace_directory_available
memory_directory_available
source_documents_present
```

Prediction should compare expected features against observed features.

This is better than treating raw text as the entire signal.

---

## Finite state machines

Finite state machines may help represent project phase, runtime readiness, or capability state.

Example project phases:

```text
unknown
source_review
prototype_spike_v0_1
prototype_spike_v0_2
foundation_restart
handoff_document_pack
restart_implementation_planning
corrected_runtime_build
```

Example capability states:

```text
disabled
permission_required
enabled_read_only
enabled_with_review
enabled_for_execution
blocked
```

Finite state machines are useful because they make transitions explicit.

They can prevent unsafe jumps.

---

## State transition rules

State transition rules define what changes are allowed.

Example:

```text
current_phase: handoff_document_pack
signal: request for next document
evidence: high
transition: active_task changes to requested document
```

Another example:

```text
capability: CameraSignalSource
current_state: disabled
signal: user mentions camera conceptually
transition: no change
reason: explicit permission not given
```

State transitions should be evidence-backed and traceable.

---

## Constraint checking

Constraint checking is central to safety.

Constraints may include:

```text
Source Documents/ is read-only.
Manual text is one signal source only.
Prediction must precede signal.
No autonomous terminal actions.
No camera/mic/window access without explicit permission.
External action requires safety check.
LLM output is proposal, not authority.
```

A constraint checker can block unsafe or architecturally invalid behavior.

Example:

```text
if action_target starts with Source Documents/ and action_type is write/delete:
  block action
```

---

## Schema validation

Schema validation helps ensure records keep a stable shape.

v0.2 added lightweight JSON Schema validation.

This remains useful later.

But schema validation should come after the corrected record model stabilizes.

Validate:

```text
base fields
record_type
required stable fields
loop order fields
source_refs
uncertainty
```

Do not overbuild schemas too early.

Schemas should protect architecture, not freeze mistakes.

---

## Event sourcing

Event sourcing means current state can be reconstructed from a sequence of events.

The Bounded Predictive Machine has a similar memory philosophy.

A cognitive tick creates trace records:

```text
body state
priors
world model
prediction
signal
prediction error
evidence
update/no-update
action/no-action
safety
outcome
memory
```

The current world model should be explainable from prior records and updates.

This supports audit, rollback, and correction.

---

## Append-only traces

Append-only traces are useful for early memory.

JSONL is a simple append-only format.

Benefits:

```text
simple
inspectable
local
easy to test
easy to diff
easy to replay later
```

Limitations:

```text
retrieval is limited
large histories become unwieldy
conflict handling is manual
staleness must be tracked separately
```

Append-only traces are a good early method, not the final memory system.

---

## Keyword retrieval

v0.2 used simple local keyword retrieval.

This is useful as an early non-LLM memory tool.

It can retrieve prior traces by matching:

```text
payload
summary
update target
update summary
record type
matched fields
matched tokens
```

Keyword retrieval is transparent.

But it is shallow.

It should be treated as contextual signal, not truth.

---

## Ranked retrieval

Ranked retrieval improves basic search by ordering results.

Simple ranking methods:

```text
number of matched tokens
matched field importance
record type priority
recency
source reliability
staleness penalty
```

This can be implemented without embeddings.

Ranked retrieval should still preserve uncertainty.

The top result is not automatically correct.

---

## Recency and staleness weighting

Memory should consider time.

A recent direct human correction may be stronger than an older summary.

Possible staleness features:

```text
created_at
model_version
current phase mismatch
superseded_by later update
conflicting newer memory
```

Staleness weighting can improve retrieval and evidence quality.

It should be transparent.

---

## Source reliability weighting

Different sources have different reliability for different claims.

Examples:

```text
human text: high for user intention
system status: high for narrow runtime facts
Git status: high for repo state if command is approved
memory trace: medium context
LLM output: proposal only
ambiguous text: low
```

Source reliability should be scoped.

A source reliable for one kind of claim may be unreliable for another.

---

## Confidence scoring

Confidence scoring may be useful, but it should be transparent.

Early scoring can use labels:

```text
low
medium
high
```

Later scoring may use numeric values.

But scores should include reasons:

```text
base score
source reliability
prediction error type
uncertainty penalty
boundary penalty
staleness penalty
```

Avoid hidden confidence numbers.

---

## Thresholds

Thresholds can decide whether updates or actions are allowed.

Example:

```text
update_threshold = 0.6
action_threshold = 0.8
sensitive_action = human_review_required
```

Thresholds should be explicit.

They should appear in records when used.

Different actions should have different thresholds.

Updating the world model and mutating files should not require the same evidence level.

---

## Bayesian updating

Bayesian updating may be useful later for belief calibration.

It can help represent how evidence changes confidence.

Possible use:

```text
prior confidence
likelihood of signal given model
likelihood of signal given alternative model
posterior confidence
```

However, Bayesian methods should not be added before the corrected record order is stable.

Bayesian math should be used only where the assumptions are clear.

Do not add fake precision.

---

## Probabilistic graphical models

Probabilistic graphical models may eventually represent dependencies among:

```text
signal source reliability
prediction accuracy
evidence quality
world model confidence
action risk
outcome likelihood
```

This is a later-stage tool.

It should not be part of the first restart.

The first restart should use simple deterministic records.

---

## Anomaly detection

Anomaly detection may help identify unexpected signals or runtime states.

Examples:

```text
unexpected signal source
unexpected file change
unexpected resource load
unexpected repeated prediction failure
unexpected action request
unexpected memory conflict
```

Early anomaly detection can be rule-based.

Later it could use statistical methods.

Anomaly detection should increase uncertainty, not automatically trigger action.

---

## Classifiers

Classifiers may eventually identify signal features.

Examples:

```text
is this manual text a correction?
is this a document request?
is this an unsafe action request?
is this a clarification request?
is this a system error?
```

Early classifiers can be deterministic rules.

Later classifiers may be learned or LLM-assisted.

Classifier outputs should be signals/proposals, not authority.

---

## Symbolic logic

Symbolic logic may help enforce constraints.

Examples:

```text
If source is disabled, signal is inadmissible.
If action target is read-only, write action is blocked.
If evidence quality is low, model update is not allowed.
If action has no expected effect, action proposal is invalid.
```

Symbolic rules are useful because they are explicit and testable.

They fit the early architecture well.

## Planning methods

Planning may be useful later, but not at the foundation.

When planning is introduced, it should be downstream of:

```text
world model
prediction
evidence
safety
capability admission
```

Planning should not become an agent loop that bypasses prediction and evidence.

Future planning tools might include:

```text
simple task graphs
state machines
goal decomposition
constraint-based planning
human-approved action plans
```

Planning must remain bounded.

## Control theory ideas

The system has control-like aspects:

```text
expected state
observed state
error
correction
feedback
stability
```

Control ideas may help with:

```text
runtime health
resource load
prediction calibration
action outcome monitoring
error reduction
```

But BPM is broader than a normal controller because it handles semantic signals, memory, model revision, and safety.

Control theory can contribute tools, not define the whole system.

---

## Active inference ideas

Active inference may provide useful inspiration for action as future uncertainty or error management.

Possible useful idea:

```text
action can be chosen to improve future evidence or reduce uncertainty
```

But BPM must remain safety-governed.

The system should not simply act to reduce error.

It must ask:

```text
Is the action allowed?
Is it bounded?
Is the expected effect observable?
Is no-action safer?
Is human review required?
```

Active inference is inspiration, not permission for autonomous action.

Predictive processing is a conceptual inspiration.

Useful pattern:


## Vector search and embeddings

Embeddings may eventually improve retrieval.

Useful role:

```text
find semantically similar memory traces or documents
```

Unsafe role:

```text
decide truth
own belief
authorize updates
authorize action
```

Embeddings should be retrieval tools only.

They should produce candidate context, not conclusions.

Do not add embeddings before the corrected loop is stable.

## Test-driven development

Test-driven development is one of the strongest non-LLM tools in this project.

Tests should enforce architecture.

Important tests:

```text
prediction before signal
manual text is one signal source only
prediction does not copy input
prediction error exists before evidence
evidence exists before update
low evidence produces NoUpdate
unsafe action produces NoAction or blocked safety
memory trace references source records
loop order is preserved
```

Tests protect the project from drifting back to manual-text-first design.

---

## Property-based testing

Property-based testing may be useful later.

It can test invariants across many generated inputs.

Possible invariants:

```text
SignalRecord never appears before PredictionRecord in a valid loop.
Low evidence never produces external action.
Disabled signal sources never produce admissible signals.
Source Documents/ write actions are always blocked by default.
NoUpdate never changes world model state.
```

This is a future improvement, not required in the first restart.

# Tool selection principles

When considering a new method or tool, ask:

```text
Does this preserve the corrected loop order?
Does it keep prediction before signal interpretation?
Does it treat signals as observations, not truth?
Does it preserve evidence gating?
Does it preserve NoUpdate and NoAction?
Does it keep safety downstream and structural?
Does it preserve traceability?
Does it avoid making an LLM the core mind?
Does it avoid hidden authority?
Does it make uncertainty visible?
```

If the answer is no, do not add the tool yet.

# How LLM assistance could be safely introduced later

If an LLM is introduced later, it should be wrapped as a signal-producing component.

Example:

```text
LLMInterpretationProposalSource
```

It may produce:

```text
candidate_interpretation
alternative_interpretations
ambiguity_flags
summary_proposal
```

The runtime should then check:

```text
source = LLM proposal
confidence = proposal only
requires evidence check = true
can update model directly = false
can authorize action = false
```

The LLM should not write directly to the world model.

---

## LLM output as signal

Treat LLM output like any other signal:

```text
source
payload
observed_features
uncertainty
admissibility
prediction comparison
evidence quality
```

An LLM may help propose meaning.

But the runtime decides whether that proposal is admissible and useful.

---

## LLM output and memory

LLM-generated summaries can be helpful, but dangerous if treated as memory authority.

A summary should preserve source refs.

Bad:

```text
LLM summary becomes memory truth.
```

Better:

```text
LLM summary is stored as a derived view linked to source records.
```

Best:

```text
Memory trace remains authoritative; LLM summary is a human-readable rendering.
```

---

## LLM output and action

LLMs should not authorize action.

If an LLM proposes an action, it should become an `ActionProposalRecord` candidate only if the runtime accepts it as a proposal.

It must still pass:

```text
evidence gate
boundary check
permission check
safety check
human review if needed
```

# Common failure modes

## Failure mode 1: LLM becomes hidden controller

The system calls an LLM and follows its output as if it were the runtime.

This breaks the architecture.

---

## Failure mode 2: embeddings become truth

The system retrieves similar memories and treats them as correct.

This bypasses evidence.

---

## Failure mode 3: scoring becomes fake precision

The system produces numeric confidence without justified assumptions.

Transparent labels may be better early.

---

## Failure mode 4: tools arrive before boundaries

The system adds file, terminal, network, or sensor tools before safety is ready.

This creates unsafe capability.

---

## Failure mode 5: complexity hides the loop

The architecture becomes harder to inspect because advanced methods obscure the basic tick.

The loop must remain visible.

---

## Failure mode 6: memory retrieval replaces world model

The system uses retrieval results instead of maintaining a structured world model.

Retrieval should support the model, not replace it.

---

# Design principles

Future methods and tools should preserve these principles:

```text
1. The runtime owns the loop.
2. LLMs may propose, not decide.
3. Embeddings may retrieve, not judge truth.
4. Rules should be transparent before methods become complex.
5. Prediction must remain model-generated.
6. Signals must remain bounded observations.
7. Prediction error must remain explicit.
8. Evidence quality must gate update.
9. Safety must gate action.
10. Memory must preserve source-backed reconstruction.
11. Tools must be admitted before use.
12. Sensitive sensors require explicit permission.
13. NoUpdate and NoAction remain valid outcomes.
14. Tests should protect architectural invariants.
15. Complexity should be added only when the simpler system is stable.
```
