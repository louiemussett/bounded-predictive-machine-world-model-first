# 02 — Corrected Architecture Overview

## Purpose of this document

This document explains the corrected architecture for the **Bounded Predictive Machine** after the v0.1/v0.2 learning spikes.

Its purpose is to make the foundational loop impossible to misunderstand.

The key correction is:

```text
The system must begin from body state, priors, and a world model.
It must generate prediction before receiving or interpreting new signals.
Manual text is only one signal source.
```

This document should be read after:

```text
00-project-continuity-brief.md
01-core-vision-and-thesis.md
```

and before any new implementation work begins.

---

## The architectural correction in one sentence

The Bounded Predictive Machine must not start from manual text input; it must start from body/system state, priors, and a world model, generate predictions from that model, then compare incoming signals against those predictions to produce prediction error, update, abstention, or action.

---

## The old loop shape

The v0.1/v0.2 prototype effectively used this loop:

```text
manual text input
→ prediction
→ signal
→ interpretation
→ evidence quality
→ belief update or no-update
→ action or no-action
→ safety
→ outcome
→ memory
```

This loop was useful as a prototype spike.

It proved that the runtime could create records, link traces, abstain, score evidence, choose no-action, persist JSONL records, validate schemas, retrieve local memory, and expose behavior through a CLI.

But this loop encoded the wrong foundation.

It made manual text the practical starting point of cognition.

That is not the intended architecture.

---

## Why the old loop is insufficient

The old loop has several problems.

### 1. Manual text becomes the root

The prototype treated manual user input as the first thing that happens.

That makes the system resemble a structured text-processing pipeline.

But the intended system is not a chatbot and not a text-first assistant.

Manual text should be only one signal source among many.

---

### 2. Prediction becomes shallow

If the system starts from manual text, prediction can become too close to the input.

The bad pattern is:

```text
user says X
→ system predicts something about X
→ system observes X
```

That is circular.

It does not prove that the system can predict from an internal model.

---

### 3. The world model is missing or secondary

The intended architecture requires the system to maintain a model of itself and its world before signals arrive.

The old loop did not make this primary enough.

It had records and traces, but it did not begin from a structured body state, prior model, and world model.

---

### 4. Signal ecology is too narrow

The old loop mainly handled manual text.

The intended system should eventually support signals from many bounded sources:

```text
manual text
system status
filesystem state
process state
terminal observations
active window/application state
camera input
microphone input
memory traces
internal runtime state
network state, if explicitly allowed later
```

The architecture must therefore be signal-source agnostic from the beginning.

---

### 5. Future layers would inherit the wrong shape

If memory, embeddings, Bayesian scoring, action selection, sensors, or LLM proposal tools are added on top of a manual-text-first loop, then the whole system may become a more elaborate chatbot pipeline.

The foundation must be corrected before expansion.

---

## The corrected loop shape

The corrected architecture is:

```text
body/system state + priors + world model
→ prediction
→ multi-source signal intake
→ prediction comparison
→ prediction error
→ evidence quality
→ world-model / belief update or no-update
→ action / no-action
→ safety
→ outcome
→ memory
→ revised model
```

This is the loop that future documentation, implementation, and Codex prompts must preserve.

---

## Corrected loop in expanded form

A more explicit version is:

```text
1. Read current body/system state
2. Load priors and structural assumptions
3. Load current world model
4. Generate prediction from body + priors + world model
5. Select or receive bounded signal source
6. Capture signal
7. Compare observed signal against prediction
8. Produce prediction error
9. Evaluate evidence quality
10. Decide update, no-update, or gather-more-signal
11. Propose action or no-action
12. Run safety checks
13. Observe outcome if action occurs
14. Write memory trace
15. Revise model or preserve uncertainty
16. Prepare next prediction
```

The order matters.

Prediction must happen before signal interpretation.

Signal must be compared against prediction.

Update must depend on evidence quality.

Action must depend on safety.

Memory must preserve the chain.

---

## The core dependency order

The corrected architecture depends on this ordering:

```text
BodyStateRecord
PriorModelRecord
WorldModelRecord
→ PredictionRecord
→ SignalSourceRecord
→ SignalRecord
→ PredictionErrorRecord
→ EvidenceRecord
→ ModelUpdateRecord or NoUpdateRecord
→ ActionProposalRecord or NoActionRecord
→ SafetyCheckRecord
→ OutcomeRecord
→ MemoryTraceRecord
→ LoopRecord
```

This record order is not just a coding preference.

It expresses the cognitive thesis of the project.

The system must have some model before it predicts.

The system must predict before it evaluates a signal.

The system must compare prediction and signal before learning.

The system must evaluate evidence before updating.

The system must pass safety before action.

The system must preserve trace before continuity.

---

## The corrected architecture as a stack

The corrected system can be described as a stack.

```text
Layer 0 — Safety Shell
boundary, permissions, capability admission, read-only zones, recovery, audit

Layer 1 — Body State
runtime identity, current system state, resources, allowed sensors, allowed actions

Layer 2 — Priors
structural assumptions, default uncertainty rules, safety biases, expected noise

Layer 3 — World Model
current model of task, environment, user context, system state, constraints

Layer 4 — Prediction
model-generated expectation about future signals or state transitions

Layer 5 — Signal Sources
manual text, system status, files, terminal observations, memory, future sensors

Layer 6 — Signal Records
bounded captured observations with source, payload, admissibility, uncertainty

Layer 7 — Prediction Error
comparison between expected and observed signal features

Layer 8 — Evidence Quality
judgment of whether the signal/error is reliable enough to matter

Layer 9 — Model Update / NoUpdate
revision or preservation of world model based on evidence

Layer 10 — Action / NoAction
bounded proposal, abstention, or no-action decision

Layer 11 — Outcome
observed result or explicit lack of external execution

Layer 12 — Memory
reconstructive trace of the full chain

Layer 13 — Metacognitive Readiness
remaining uncertainty, confidence, conflicts, readiness for next tick
```

This stack should guide future implementation.

---

## Model first, not input first

The most important design rule is:

```text
Prediction must be generated from model state, not copied from incoming input.
```

Bad pattern:

```text
manual input says X
→ prediction expects X
→ signal confirms X
```

Correct pattern:

```text
world model expects X
→ signal arrives showing Y
→ compare X and Y
→ produce prediction error
→ update, no-update, or gather more evidence
```

The corrected system should be able to say:

```text
Before the signal arrived, my model expected X.
The signal showed Y.
The comparison produced this error.
The evidence was this strong.
Therefore I updated, did not update, or need more evidence.
```

That is the core of the architecture.

---

## Body state

Body state is the system’s current operational condition.

For this project, body state may include:

```text
- runtime identity
- current project path
- allowed directories
- read-only directories
- current process state
- current resource state
- available signal sources
- available action capabilities
- enabled or disabled sensors
- safety mode
- current trace location
- current memory location
```

The system cannot responsibly predict or act without knowing its own body and boundary.

A machine that does not know what it can access, affect, or observe should not be given broad autonomy.

---

## Priors

Priors are structural assumptions that exist before a new signal arrives.

Examples:

```text
- signals may be incomplete
- user text may be ambiguous
- memory retrieval is context, not truth
- source documents are read-only unless explicitly allowed
- external actions require safety checks
- sensors require explicit permission
- low evidence should produce no-update
- unclear action targets should produce no-action
- prediction mismatch is not automatically proof
```

Priors help the system generate predictions and avoid unsafe overreaction.

They are not fixed forever.

They can be revised later if the system has enough evidence and permission to revise them.

---

## World model

The world model is the system’s current structured understanding of its environment and task context.

It may represent:

```text
- current project phase
- current user intention
- active constraints
- known files and documents
- known unresolved questions
- known system capabilities
- known risks
- expected next signals
- relevant memory traces
- confidence levels
```

The world model should not be a prose summary only.

It should be structured enough to generate predictions and support traceable revision.

---

## Prediction

Prediction is a model-generated expectation about what may happen next.

A prediction may target:

```text
- a signal source
- a signal type
- signal features
- system state
- user behavior
- file state
- runtime condition
- action outcome
- uncertainty state
```

A useful prediction should include:

```text
- prediction target
- expected value or feature
- match conditions
- source model references
- confidence
- uncertainty
```

The prediction does not need to be perfect.

It needs to be explicit enough to test.

---

## Signal source

A signal source is a bounded channel from which observations may arrive.

Examples:

```text
ManualTextSignalSource
SystemStatusSignalSource
FileStateSignalSource
TerminalObservationSignalSource
MemoryTraceSignalSource
WindowStateSignalSource
CameraSignalSource
MicrophoneSignalSource
```

Early restart implementation should only use safe sources such as:

```text
ManualTextSignalSource
SystemStatusSignalSource
possibly ProjectDirectorySignalSource
```

Camera, microphone, window state, terminal actions, and network access should not be added until explicit boundaries and permissions exist.

---

## Signal

A signal is an observation captured from a signal source.

A signal should include:

```text
- source
- payload or observed features
- timestamp
- boundary status
- admissibility status
- source reliability, if known
- uncertainty
```

Signals are not truth.

Signals are evidence candidates.

They must be interpreted in relation to prediction, source, boundary, and reliability.

---

## Prediction comparison

The system must compare prediction and signal.

The comparison asks:

```text
What did the model expect?
What did the signal show?
Which expected features matched?
Which expected features failed?
Which features were missing?
Was the signal from an expected source?
Was the signal admissible?
Was the result inconclusive?
```

This comparison creates prediction error.

---

## Prediction error

Prediction error is the structured result of comparing expected and observed features.

Possible prediction error states include:

```text
match
mismatch
partial_match
inconclusive
missing_signal
unexpected_signal_source
unexpected_signal_type
contradiction
```

Prediction error creates pressure to revise the model.

But prediction error alone does not justify revision.

Evidence quality must still be checked.

---

## Evidence quality

Evidence quality determines whether a signal and prediction error are strong enough to support update.

Evidence quality may consider:

```text
- signal source reliability
- signal clarity
- ambiguity
- freshness
- consistency with other records
- severity of mismatch
- number of uncertainty flags
- whether the source is allowed
- whether the observation is repeatable
```

The system should not update simply because something unexpected happened.

A weak mismatch may produce no-update.

A strong mismatch may justify model revision.

A dangerous or uncertain mismatch may require more observation or human review.

---

## Update, no-update, or gather more signal

After evidence quality is assessed, the system should choose among at least three outcomes:

```text
ModelUpdate
NoUpdate
GatherMoreSignal
```

### ModelUpdate

Used when evidence is strong enough to revise the world model or belief state.

### NoUpdate

Used when the signal is too weak, ambiguous, inadmissible, or unclear.

### GatherMoreSignal

Used when the signal may matter but the system needs additional observation before changing the model.

No-update and gather-more-signal are not failures.

They are core intelligent outcomes.

---

## Action and no-action

Action should not be obedience.

Action should be a bounded proposal or intervention expected to affect future signals, uncertainty, model quality, or system state.

Before action, the system should ask:

```text
What action is being proposed?
Why is it being proposed?
What effect is expected?
Is the effect observable?
Is the action inside boundary?
Is permission present?
What could go wrong?
Can the action be reversed?
Is no-action safer?
```

No-action is valid when:

```text
- evidence is weak
- permission is missing
- target is unclear
- boundary is uncertain
- action effect cannot be predicted
- outcome cannot be observed
- safety check fails
```

The system may observe before acting.

The system may propose before executing.

---

## Safety

Safety governs capability.

A capability should not be available merely because the machine can technically perform it.

It must be:

```text
admitted
scoped
bounded
traceable
revocable
recoverable where possible
```

Early restart safety boundaries should include:

```text
- no autonomous filesystem mutation
- no external network access unless explicitly allowed
- no camera/mic/window access without explicit permission
- terminal/system access observation-only at first
- Source Documents remain read-only unless explicitly changed
- no LLM as action authority
- human approval required for risky capabilities
```

Safety is part of the architecture, not an add-on.

---

## Outcome

An outcome records what happened after an action or no-action decision.

Outcomes should include:

```text
- selected action or no-action reference
- expected effect
- observed effect
- success or failure
- unresolved questions
- uncertainty
```

If no external action occurred, that should be recorded explicitly.

The system should not pretend an action succeeded if it only proposed the action.

---

## Memory

Memory should reconstruct the cognitive chain.

A memory trace should link:

```text
body state
priors
world model
prediction
signal source
signal
prediction error
evidence quality
update/no-update
action/no-action
safety
outcome
remaining uncertainty
```

Memory should not be a loose summary.

It should preserve the evidence trail that explains why the system changed or did not change.

Retrieved memory is context, not truth.

---

## Revised model

At the end of a loop, the system should have either:

```text
- a revised world model
- an unchanged world model with uncertainty preserved
- a request for more signal
- a blocked action
- a safe no-action result
```

The revised or preserved model becomes the starting point for the next prediction.

This creates continuity.

---

## The cognitive tick

The corrected loop can be treated as one **cognitive tick**.

A cognitive tick is one complete cycle of:

```text
model
→ prediction
→ signal
→ error
→ evidence
→ update/no-update
→ action/no-action
→ safety
→ outcome
→ memory
→ revised model
```

The cognitive tick is not a chat turn.

It is not a tool call.

It is not a model response.

It is a complete traceable unit of machine sense-checking.

---

## Minimal corrected prototype

The first corrected implementation should not be impressive.

It should only prove the correct order.

Minimum target:

```text
Create BodyStateRecord
Create PriorModelRecord
Create WorldModelRecord
Generate PredictionRecord from those records
Capture SignalRecord from one safe SignalSource
Compare prediction and signal
Create PredictionErrorRecord
Evaluate EvidenceRecord
Create ModelUpdateRecord or NoUpdateRecord
Create ActionProposalRecord or NoActionRecord
Create SafetyCheckRecord
Create OutcomeRecord
Create MemoryTraceRecord
Create LoopRecord preserving order
```

Do not add broad sensors yet.

Do not add embeddings yet.

Do not add autonomous terminal actions yet.

Do not make an LLM the cognitive authority.

---

## First safe signal sources

The corrected restart should begin with safe and simple signal sources.

Recommended first sources:

```text
ManualTextSignalSource
SystemStatusSignalSource
ProjectDirectorySignalSource, if read-only and bounded
```

Manual text is still useful for tests.

But it must be represented as one source among others.

It must not be the root of the loop.

## Example corrected tick

Example world model:

```text
current_project_phase = "architecture correction"
expected_next_signal_type = "restart planning or foundation clarification"
manual_text_available = true
system_status_available = true
confidence = medium
```

Prediction:

```text
The next relevant signal may concern corrected foundation, restart planning, or objection to manual-text-first design.
```

Signal:

```text
manual_text: "we need to start from scratch"
```

Prediction error:

```text
partial_match / match
The signal matches the predicted project phase and restart-planning expectation.
```

Evidence quality:

```text
medium/high, because the signal is clear and directly relevant.
```

Model update:

```text
Update current_project_phase to "foundation restart".
```

Action:

```text
No external action.
Propose writing corrected architecture documents.
```

Memory:

```text
Record that the model shifted from continuing v0.2 to restarting from corrected foundation.
```

---

## Example no-update tick

Example world model:

```text
current_project_phase = "architecture correction"
expected_next_signal_type = "specific restart instruction"
```

Prediction:

```text
The next signal should contain a clear instruction about restart direction.
```

Signal:

```text
manual_text: "???"
```

Prediction error:

```text
mismatch / inconclusive
The signal does not contain enough information to match the expected instruction.
```

Evidence quality:

```text
low
```

Model update:

```text
NoUpdate
```

Action:

```text
NoAction or GatherMoreSignal
```

Memory:

```text
Record unclear signal and preserved uncertainty.
```

This is not failure.

This is correct behavior.

## What future prompts must enforce

Every future Codex prompt should protect the corrected shape.

Prompts should say things like:

```text
Do not start the loop from manual text.
Prediction must be generated from BodyStateRecord, PriorModelRecord, and WorldModelRecord.
Signal intake must occur after prediction generation.
Manual text must be implemented only as one SignalSource.
PredictionErrorRecord must compare expected features against observed features.
No-update and no-action must remain valid outcomes.
Do not add LLM integration.
Do not add embeddings yet.
Do not add broad sensors yet.
Do not add autonomous actions yet.
```

This prevents the implementation from drifting back into the v0.2 shape.

---

## What should not happen next

The next implementation should not jump to:

```text
- embeddings
- vector memory
- Bayesian scoring
- camera input
- microphone input
- active window tracking
- autonomous terminal actions
- broad filesystem access
- LLM interpretation
- agent planning
- tool orchestration
- UI polish
```

Those may be useful later.

But they should not precede the corrected foundation.

---

## What should happen next

The correct next sequence is:

```text
1. Finish the handoff document pack.
2. Create a clean restart branch.
3. Implement minimal world-model-first records.
4. Implement model-generated prediction.
5. Implement signal source abstraction.
6. Implement manual text and system status as safe signal sources.
7. Implement prediction error comparison.
8. Implement update/no-update.
9. Implement no-action and safety check.
10. Implement memory trace and loop record.
11. Only then consider CLI, persistence, retrieval, or more sensors.
```

The documentation should come first because the architecture must be preserved before code begins.

## Final summary

The corrected architecture is not:

```text
manual text → prediction → response
```

It is:

```text
body/system state + priors + world model
→ prediction
→ signal
→ prediction error
→ evidence quality
→ update/no-update
→ action/no-action
→ safety
→ outcome
→ memory
→ revised model
```

The Bounded Predictive Machine must be world-model-first, signal-source-agnostic, evidence-gated, uncertainty-aware, safety-governed, and trace-backed from the beginning.

That is the corrected architecture.
