# 03 — Lessons from v0.1 to v0.2

## Purpose of this document

This document preserves the lessons learned from the first two prototype spikes of the **Bounded Predictive Machine** project.

The goal is not to dismiss v0.1 and v0.2.

The goal is to understand exactly what they proved, what they failed to prove, and what must be carried forward into the corrected restart.

The most important conclusion is:

```text
v0.1 and v0.2 were useful learning spikes, not the final foundation.
```

They produced valuable runtime mechanics, tests, records, traces, and implementation habits.

But they also revealed that the foundational loop was shaped incorrectly because it began from manual text input instead of body state, priors, and a world model.

---

## Project context

The project is the **Bounded Predictive Machine**.

Short form:

```text
BPM
```

The long-term goal is to build a bounded predictive cognitive runtime that starts from:

```text
body/system state + priors + world model
```

then generates prediction, observes signals, compares prediction against observation, computes prediction error, updates or abstains based on evidence, and acts only within explicit safety boundaries.

The v0.1/v0.2 prototypes did not fully implement this corrected architecture.

They implemented a narrower manual-text-first loop that helped reveal what the real foundation needed to be.

---

## Summary of v0.1

v0.1 was the first build spike.

It established the initial Python project structure and proved that a simple record-backed runtime loop could be built in small, testable chunks.

The v0.1 tag was:

```text
v0.1-first-loop
```

The early implementation introduced concepts such as:

```text
- record classes
- trace writing
- boundary/path checks
- safety checks
- manual text signal wrapping
- deterministic prediction creation
- deterministic signal interpretation
- evidence quality
- scoped belief update
- action candidate / no-action creation
- outcome records
- memory traces
- loop record linking
- manual text loop orchestration
- negative tests
```

v0.1 proved that the project could be built incrementally and tested continuously.

It also showed that explicit records were useful for making runtime behavior inspectable.

---

## Summary of v0.2

v0.2 extended the first prototype into a more complete non-LLM cognitive runtime spike.

The branch was:

```text
spike/v0.2-non-llm-cognitive-runtime
```

The v0.2 tag was:

```text
v0.2-non-llm-cognitive-runtime
```

v0.2 added several important mechanisms:

```text
- run-once CLI
- readable loop summaries
- optional JSONL persistence
- ignored generated trace outputs
- lightweight JSON Schema validation
- uncertainty gate records
- abstention behavior
- empty input handling
- transparent evidence and belief scoring
- local JSONL memory retrieval
- memory retrieval integrated into the manual loop
- capped and ranked retrieval
- spike closeout documentation
```

v0.2 showed that many non-LLM mechanisms could support cognition-like runtime discipline.

It also showed that the architecture could abstain, choose no-action, preserve uncertainty, and explain its record chain.

---

## What v0.1 and v0.2 proved

v0.1 and v0.2 proved several valuable things.

These should be preserved conceptually in the restart.

---

### 1. Test-first incremental development works

The project progressed well when each implementation chunk was small, specific, and testable.

The working pattern was:

```text
Goal
Scope
Allowed files
Requirements
Non-goals
Tests
Summary after editing
```

This style should be preserved for future Codex prompts.

The prompt discipline was good.

The architectural root needs correction.

---

### 2. Records make the runtime inspectable

The project benefited from explicit records such as:

```text
PredictionRecord
SignalRecord
InterpretationRecord
EvidenceQualityRecord
BeliefUpdateRecord
NoUpdateRecord
ActionCandidateRecord
NoActionRecord
SafetyCheckRecord
OutcomeRecord
MemoryTraceRecord
LoopRecord
```

Records made it possible to inspect what happened in a loop.

They allowed the system to show:

```text
what was predicted
what signal arrived
how it was interpreted
how evidence was scored
what belief changed or did not change
what action was proposed or refused
what safety decided
what outcome occurred
what memory trace was written
```

This is a strong pattern and should continue.

---

### 3. Serialization is useful

The `to_dict()` and `to_json()` pattern was useful because it made records portable, inspectable, and testable.

Future records should continue to serialize cleanly.

The restart should preserve:

```text
to_dict()
to_json()
JSON-compatible record fields
stable record_type values
source_refs
loop_id
uncertainty
```

---

### 4. JSONL traces are useful

JSONL persistence was useful for recording loop history and enabling simple local memory retrieval.

Generated traces should remain ignored by Git unless intentionally saved as examples.

The restart can reuse the idea of JSONL traces later, but persistence should not be added before the corrected loop shape is implemented.

Correct order:

```text
correct loop shape first
persistence second
retrieval later
```

---

### 5. Schema validation is useful

The lightweight JSON Schema validation added in v0.2 helped keep runtime records disciplined.

It validated base fields such as:

```text
id
record_type
created_at
created_by
loop_id
status
source_refs
uncertainty
```

This was useful and should be carried forward after the restart records stabilize.

But schema validation should not drive the architecture.

The corrected record model must come first.

---

### 6. Abstention is essential

v0.2 demonstrated that the system can explicitly abstain when evidence is insufficient.

Examples:

```text
empty input → abstention
unclear input → abstention
low evidence score → no update
```

This is one of the strongest lessons from v0.2.

The restart should preserve abstention as a first-class outcome.

A system that can say “not enough evidence” is safer and more honest than one that always updates or acts.

---

### 7. NoUpdate is essential

The system should not change its model merely because a signal arrived.

v0.2 proved that a loop could create a structured no-update result when evidence was weak.

This is important for the corrected architecture.

Prediction error should create update pressure, but evidence quality should decide whether update is justified.

NoUpdate must remain valid.

---

### 8. NoAction is essential

v0.2 showed that the system can produce no-action when action is not justified.

This should remain a core design principle.

NoAction is appropriate when:

```text
- evidence is weak
- signal is ambiguous
- action target is unclear
- safety is uncertain
- permission is missing
- external effect is not justified
```

NoAction is not failure.

It is part of intelligent bounded behavior.

---

### 9. Safety can be deterministic and inspectable

The first safety checks were minimal, but they established an important pattern:

```text
action proposal
→ safety check
→ allowed or blocked
→ outcome record
```

This should be preserved.

The restart should make safety more foundational, especially for signal sources, system access, and external actions.

Safety should govern capability admission, not merely approve actions after the fact.

---

### 10. Local memory retrieval can work without an LLM

v0.2 implemented keyword-style local JSONL memory retrieval.

This showed that memory support does not require an LLM or embeddings at the first stage.

The retrieval system could find prior records based on matched tokens and fields.

This was useful, but it should not be added too early in the restart.

Memory retrieval should come after the corrected world-model-first loop exists.

---

### 11. CLI observability is helpful

The CLI summaries were useful because they made the loop easy to inspect.

For example, the summary format could show:

```text
Prediction
Signal
Memory retrieval
Interpretation
Evidence quality
Evidence score
Uncertainty gate
Belief result
Abstention
Action/no-action
Safety result
Outcome
Memory trace
Remaining uncertainty
```

This made development easier.

However, CLI polish should not take priority over the corrected architecture.

The restart should implement the loop first, then add CLI visibility.

---

### 12. Codex prompt discipline worked

The prompts used for Codex were scoped and effective.

They typically included:

```text
- Proceed with the next implementation chunk only
- Clear goal
- Allowed files
- Requirements
- Non-goals
- Dependency constraints
- Tests
- Do not commit
- Summarize changes
```

This method should continue.

But every restart prompt must now enforce:

```text
model-first loop
prediction before signal
manual text as one signal source only
prediction error from expected-vs-observed comparison
no LLM as cognitive authority
no broad sensors before safety boundaries
```

---

## What v0.1 and v0.2 did not prove

The prototypes did not prove the full intended architecture.

They did not prove that the system can generate predictions from a world model before observing signals.

They did not prove that the system can handle a true multi-source signal ecology.

They did not prove world-model revision from prediction error.

They did not prove body-state-driven prediction.

They did not prove sensor-source abstraction.

They did not prove safe capability expansion.

They were learning spikes.

Useful, but not foundational.

---

## The major architectural mistake

The central mistake was:

```text
manual text input became the beginning of the loop
```

The implemented shape was effectively:

```text
manual text input
→ prediction
→ signal
→ interpretation
→ evidence quality
→ belief update / no-update
→ action / no-action
→ safety
→ outcome
→ memory
```

But the intended shape is:

```text
body/system state + priors + world model
→ prediction
→ multi-source signal intake
→ prediction comparison
→ prediction error
→ evidence quality
→ world-model / belief update or no-update
→ action / no-action
→ safety
→ outcome
→ memory
→ revised model
```

The difference is not cosmetic.

It changes the identity of the system.

---

## Why manual-text-first is dangerous

Manual text is useful for testing, but dangerous as a foundation.

If manual text is the root, the system becomes:

```text
user input processor
```

rather than:

```text
world-model-first predictive runtime
```

A manual-text-first system tends to ask:

```text
What did the user say?
How should I interpret it?
Should I respond?
```

The intended system should ask:

```text
What is my current model?
What do I expect?
What signal arrived?
Did it match expectation?
Does the model need revision?
Is action justified?
```

That is the essential correction.

---

## The prediction problem in v0.1/v0.2

The prototype prediction was too shallow.

It often looked like:

```text
manual text signal may contain feedback, clarification, or an update-relevant observation
```

This is a useful placeholder, but not a true model-based prediction.

A stronger prediction should come from a structured world model.

Example:

```text
World model:
- current project phase is architecture correction
- user is evaluating whether v0.2 matches intended architecture
- previous signal indicated concern about manual-text-first design

Prediction:
- next relevant signal may object to the prototype foundation
- expected source may be manual text
- expected content may concern restart, world model, or signal sources
```

That is closer to the intended architecture because prediction arises from model state, not from the input itself.

---

## The signal problem in v0.1/v0.2

The prototype treated signal mainly as wrapped manual text.

A true BPM signal system needs signal sources such as:

```text
ManualTextSignalSource
SystemStatusSignalSource
FileStateSignalSource
TerminalObservationSignalSource
ProcessStateSignalSource
MemoryTraceSignalSource
WindowStateSignalSource
CameraSignalSource
MicrophoneSignalSource
```

The restart should begin safely with only simple sources.

But the abstraction must be designed so manual text is not privileged as the cognitive root.

---

## The memory timing problem

v0.2 added memory retrieval before the world-model-first loop was corrected.

That was useful as a spike, but it should not define the restart order.

The corrected order should be:

```text
1. body state
2. priors
3. world model
4. prediction
5. signal source abstraction
6. signal capture
7. prediction error
8. evidence
9. update/no-update
10. action/no-action
11. safety
12. memory trace
13. retrieval later
```

Memory retrieval should support prediction and context later.

It should not be used to compensate for a missing world model.

---

## The scoring lesson

v0.2 introduced transparent evidence and belief scoring.

This was useful because the system could explain why it updated or abstained.

Example pattern:

```text
base_score
prediction_penalty
uncertainty_penalty
threshold
final evidence score
```

The restart should preserve scoring transparency.

But scoring should be applied to the corrected architecture.

In the restart, scoring should evaluate evidence after prediction-error comparison, not merely after interpreting manual text.

---

## The schema lesson

Schema validation was useful after records existed.

But schemas should remain lightweight.

They should not over-specify unstable fields too early.

The restart should first stabilize the corrected record set:

```text
BodyStateRecord
PriorModelRecord
WorldModelRecord
PredictionRecord
SignalSourceRecord
SignalRecord
PredictionErrorRecord
EvidenceRecord
ModelUpdateRecord
NoUpdateRecord
ActionProposalRecord
NoActionRecord
SafetyCheckRecord
OutcomeRecord
MemoryTraceRecord
LoopRecord
```

Then schema validation can be added once the shape is clearer.

---

## What should be carried forward

The following concepts should be preserved from v0.1/v0.2:

```text
- small implementation chunks
- test-first workflow
- Python dataclasses as a simple starting point
- explicit record classes
- to_dict() and to_json()
- record_type fields
- loop_id fields
- source_refs
- uncertainty fields
- JSONL traces
- schema validation after record stabilization
- abstention
- no-update
- no-action
- deterministic safety checks
- transparent evidence scoring
- memory as reconstruction
- local retrieval as a later support mechanism
- CLI summaries after core loop correctness
- Git tags and save points
```

These were strengths.

They should not be discarded.

---

## What should not be carried forward uncritically

The following assumptions should not be carried into the restart:

```text
- manual text is the start of cognition
- prediction is created after or around manual text input
- signal intake is mainly text wrapping
- interpretation is the central cognitive act
- memory retrieval belongs before world-model-first prediction
- CLI ergonomics matter before loop correctness
- schema polish matters before conceptual correctness
- embeddings should be added before prediction error is fixed
- LLM interpretation should be added before deterministic foundations exist
- broad sensors should be added before safety boundaries exist
```

These were either limitations of the spike or risks for the future.

---

## What v0.2 should be called

v0.2 should be described as:

```text
non-LLM cognitive runtime spike
```

or:

```text
manual-text-first runtime mechanics spike
```

It should not be described as the foundation of the final architecture.

It is a saved prototype that proved useful mechanics.

The corrected restart should not be called v0.3.

It should be treated as a new foundation, for example:

```text
restart/v0.1-world-model-first
```

---

## Why v0.1/v0.2 were not failures

The prototypes were not failures.

They did exactly what good prototypes often do:

```text
- made hidden assumptions visible
- proved useful mechanics
- revealed the wrong foundation
- clarified what the next foundation must enforce
```

Without v0.1/v0.2, the manual-text-first error might have remained hidden much longer.

The spikes helped expose it early.

That makes them valuable.

---

## The main lesson

The main lesson is:

```text
The first stages of the loop determine the identity of the whole system.
```

If the system starts with manual text, it becomes a text-centered assistant.

If the system starts with body state, priors, and a world model, it can become a predictive runtime.

That is why the restart must correct the first stages before adding more features.

---

## Lessons for future Codex prompts

Future Codex prompts should preserve the good structure of the old prompts while enforcing the corrected architecture.

A good restart prompt should include:

```text
Proceed with the next restart implementation chunk only.

Goal:
Implement one minimal part of the world-model-first loop.

Requirements:
- Prediction must be generated from BodyStateRecord, PriorModelRecord, and WorldModelRecord.
- Signal intake must occur after prediction generation.
- Manual text must be only one SignalSource.
- PredictionErrorRecord must compare expected features against observed features.
- NoUpdate and NoAction must remain valid outcomes.
- Do not add LLM integration.
- Do not add embeddings.
- Do not add broad sensors.
- Do not add autonomous actions.
- Do not modify Source Documents/.
- Do not commit.

Tests:
- prove order
- prove prediction is not copied from manual input
- prove signal source abstraction
- prove prediction error
- prove update/no-update behavior
```

The style of the old prompts was good.

The architecture constraints must now be stronger.

---

## Lessons for future implementation order

The corrected implementation order should be:

```text
1. Define BodyStateRecord, PriorModelRecord, and WorldModelRecord.
2. Generate PredictionRecord from those records.
3. Define SignalSourceRecord and signal source abstraction.
4. Implement safe first signal sources.
5. Capture SignalRecord after prediction.
6. Compare signal to prediction.
7. Create PredictionErrorRecord.
8. Evaluate EvidenceRecord.
9. Create ModelUpdateRecord or NoUpdateRecord.
10. Create ActionProposalRecord or NoActionRecord.
11. Run SafetyCheckRecord.
12. Create OutcomeRecord.
13. Create MemoryTraceRecord.
14. Create LoopRecord preserving the corrected order.
15. Add CLI visibility.
16. Add persistence.
17. Add retrieval.
18. Add richer scoring.
19. Add additional sensors only with safety boundaries.
```

This order is designed to prevent the restart from drifting back into v0.2’s manual-text-first shape.

---

## What to preserve emotionally and practically

The v0.1/v0.2 work should be preserved as evidence of progress.

It showed that the project can be built.

It showed that Codex can implement disciplined chunks.

It showed that tests can keep the work stable.

It showed that records and traces make the system inspectable.

It showed that abstention and no-action can be built directly into the runtime.

The restart should not feel like starting over from nothing.

It is starting over with better knowledge.

---

## Final summary

v0.1 and v0.2 were valuable because they proved the mechanics of a record-backed, non-LLM runtime:

```text
records
traces
serialization
schema validation
abstention
no-update
no-action
safety
memory retrieval
CLI inspection
tests
Git save points
```

But they also revealed the central architectural flaw:

```text
manual text input became the beginning of cognition
```

The corrected architecture must begin instead with:

```text
body/system state + priors + world model
→ prediction
→ signal
→ prediction error
```

Therefore, v0.1/v0.2 should be treated as learning spikes.

The restart should preserve their useful mechanics while correcting the foundation from the first line.
