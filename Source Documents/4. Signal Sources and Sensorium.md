# 04 — Signal Sources and Sensorium

## Purpose of this document

This document defines what a **signal** is in the Bounded Predictive Machine project and explains how the system’s **sensorium** should be designed.

The key idea is:

```text
Manual text is only one signal source.
It is not the foundation of cognition.
```

The corrected architecture must be **signal-source agnostic** from the beginning.

That means the system should be designed so future signals can come from many bounded sources, including manual text, system state, files, terminal observations, memory, and eventually sensors such as camera, microphone, or active window state.

However, broader signal access must be introduced slowly, explicitly, and safely.

## Core thesis

The Bounded Predictive Machine should not treat user text as the only input to cognition.

It should treat all inputs as **signals** from bounded sources.

A signal may come from:

```text
manual text
system status
filesystem state
terminal output
process state
memory traces
active application/window state
camera
microphone
keyboard/mouse activity
network state, if explicitly allowed later
internal runtime state
```

The system should not assume that a signal is true, complete, reliable, or actionable.

A signal is only an observation candidate.

It must be bounded, captured, compared against prediction, evaluated for evidence quality, and traced.

---

## Definition of signal

A **signal** is any bounded input, observation, event, reading, trace, or state change that the system can receive or inspect.

A signal is not automatically truth.

A signal is not automatically an instruction.

A signal is not automatically evidence strong enough to update the world model.

A signal is an observed candidate for interpretation and comparison.

The system should ask:

```text
Where did this signal come from?
Was the source allowed?
Was the signal expected?
Was the signal fresh?
Was the signal clear?
Was the signal reliable?
Was the signal inside boundary?
Did it match prediction?
Did it contradict prediction?
Is it strong enough to update the model?
Is it safe to act on?
```

---

## Definition of sensorium

The **sensorium** is the system’s set of available signal sources and the rules governing how those sources may be observed.

For humans, sensorium usually means sensory channels such as sight, sound, touch, and bodily sensation.

For this machine system, sensorium means:

```text
all bounded channels through which the runtime may observe itself, the user, the environment, project state, or prior traces
```

The machine sensorium may include:

```text
manual text input
runtime state
filesystem observations
terminal observations
process observations
system resource observations
memory observations
application/window observations
camera observations
microphone observations
```

Not all sources should be enabled at once.

A source must be explicitly admitted into the system’s boundary before use.

---

## Signal sources are capabilities

Every signal source is a capability.

Even observation can be risky.

A signal source may expose:

```text
private user information
file contents
system state
application state
camera imagery
microphone audio
location-like context
credentials or secrets
behavioral patterns
sensitive project details
```

Therefore, signal sources must be governed.

A signal source should not be available merely because the machine can technically access it.

It must be:

```text
explicitly admitted
bounded
read-only unless otherwise approved
traceable
revocable
scoped to purpose
safe to inspect
```

---

## Manual text signal source

Manual text is the simplest signal source.

It is useful for early tests because it is easy to provide, easy to inspect, and safe compared with broader sensors.

But manual text must be represented as one source among many:

```text
ManualTextSignalSource
```

It should not be treated as the root of the architecture.

Bad architecture:

```text
manual text input
→ prediction
→ signal
→ interpretation
```

Correct architecture:

```text
body state + priors + world model
→ prediction
→ signal source selection / signal intake
→ signal record
→ prediction comparison
```

Manual text can be the signal used in early tests, but prediction must still be generated before the manual text is interpreted.

---

## System status signal source

A safe early signal source is system status.

A `SystemStatusSignalSource` may observe basic runtime/system facts such as:

```text
current timestamp
operating system name
Python version
current working directory
package version
test mode or runtime mode
available configured sources
trace directory location
whether memory persistence is enabled
```

At first, this source should be read-only and minimal.

It should not inspect sensitive files, environment secrets, network state, or active user activity unless explicitly allowed later.

This source helps the runtime begin from body/system state rather than manual text.

---

## Project directory signal source

A future early source may be a read-only `ProjectDirectorySignalSource`.

It could observe bounded project facts such as:

```text
current repository root
known top-level directories
whether expected folders exist
whether Source Documents/ exists
whether docs/ exists
whether src/ exists
whether tests/ exists
current Git branch, if safe and explicitly allowed
current Git status, if safe and explicitly allowed
```

This source should not modify files.

It should not read arbitrary file contents by default.

It should not scan the whole user machine.

It should be constrained to the project root.

---

## Filesystem signal source

A broader filesystem signal source may eventually observe file state and file changes.

Potential observations:

```text
file exists
file missing
file changed
file timestamp changed
file size changed
directory changed
expected document present
unexpected file present
```

This source should initially be:

```text
read-only
project-scoped
allowlist-based
traceable
```

It should not edit, delete, move, or create files without an explicit action system and safety approval.

Filesystem observation and filesystem mutation must remain separate.

Observation is not action.

---

## Terminal observation signal source

The system may eventually observe terminal outputs or command results.

This should be treated carefully.

A `TerminalObservationSignalSource` should initially be observation-only.

Safe examples may include human-provided terminal output or explicitly approved read-only commands.

Potential observations:

```text
pytest output
Git status output
Git branch output
package version output
runtime error output
command failure output
```

Unsafe or restricted areas include:

```text
commands that mutate files
commands that install packages without approval
commands that access network
commands that reveal secrets
commands that scan the user’s full machine
commands that change system settings
commands that run destructive operations
```

The system may learn from terminal observations, but it must not gain autonomous terminal control without a separate safety design.

---

## Process state signal source

A future `ProcessStateSignalSource` may observe currently running processes or runtime process information.

This could include:

```text
current process id
whether the BPM runtime is active
whether a test process is running
basic process health
resource usage for the BPM process
```

This source can become sensitive if it observes all system processes.

Therefore, early versions should limit process observation to the BPM runtime itself or explicitly allowed processes.

---

## Resource load signal source

A future `ResourceLoadSignalSource` may observe system load.

Possible observations:

```text
CPU load
memory usage
disk usage
battery status
thermal status
runtime performance
```

These signals may help the system avoid acting when system health is poor.

Example:

```text
Prediction:
The system expects enough resources to run the next loop.

Signal:
Resource load is too high.

Prediction error:
Expected viable runtime, observed degraded runtime.

Action:
NoAction or reduce workload.
```

This source should be read-only.

---

## Memory trace signal source

Memory can also be a signal source.

A prior memory trace may inform prediction, but memory must not be treated as truth.

A `MemoryTraceSignalSource` may retrieve:

```text
previous loop records
previous predictions
previous signals
previous prediction errors
previous updates
previous no-updates
previous actions
previous uncertainty
```

Retrieved memory should be treated as context.

It can influence prediction, but it should not automatically update the world model.

Memory retrieval should be clearly marked as:

```text
contextual signal
not direct truth
not automatic belief authority
```

---

## Internal runtime signal source

The system may observe its own internal state.

Examples:

```text
current loop id
current enabled modules
current safety mode
current uncertainty state
last prediction result
last update result
last action result
current trace write status
schema validation status
```

Internal runtime state is important because the system needs to model itself.

However, internal observations can still be wrong or stale.

They should still be represented as signals and traces, not assumed to be perfect truth.

---

## Active window and application signal source

A future `WindowStateSignalSource` may observe which application, document, or window is active.

This source could be useful because it gives context about what the user is doing.

But it is sensitive.

It may expose private information.

It should not be enabled without explicit permission and clear boundaries.

Possible bounded observations, if allowed later:

```text
active application name
active project window
active document title
foreground/background state
```

Restricted observations:

```text
full screen contents
private messages
password fields
browser content
unbounded window scanning
```

This source should not be part of the first corrected implementation.

---

## Camera signal source

A future `CameraSignalSource` may observe visual input.

This is a high-sensitivity source.

It should require explicit permission, clear purpose, and strong safety boundaries.

Potential future uses:

```text
visual environment observation
object presence detection
workspace state detection
human-provided visual signal
```

Early project work should not implement camera access.

Before camera access exists, the system must have:

```text
explicit user permission
clear activation state
visible recording/observation indication
bounded capture rules
no background capture
trace of when camera was used
privacy constraints
human override
```

Camera signal must be treated as uncertain evidence, not truth.

---

## Microphone signal source

A future `MicrophoneSignalSource` may observe audio input.

This is also high-sensitivity.

Potential future uses:

```text
spoken instruction
ambient sound detection
conversation signal
speech-to-text observation
```

It should not be implemented in the early restart.

Before microphone access exists, the system must have:

```text
explicit user permission
clear activation state
no background listening
bounded capture windows
trace of microphone use
privacy constraints
human override
```

Microphone signals may be noisy, ambiguous, or misrecognized.

They require uncertainty handling.

---

## Keyboard and mouse activity signal source

A future user activity signal source may observe coarse interaction state.

Examples:

```text
user active / inactive
recent keyboard activity
recent mouse activity
focus changes
idle time
```

This can quickly become invasive.

It should only be added with explicit permission and strong minimization.

The system should prefer coarse, privacy-preserving signals over detailed behavioral capture.

---

## Network state signal source

A future network signal source may observe network state.

Examples:

```text
online/offline status
allowed endpoint availability
network error state
```

This should not include arbitrary browsing or network scanning.

Network access must be explicitly allowed and scoped.

For early work:

```text
No web access.
No autonomous network use.
No hidden external calls.
```

---

## External tool result as signal

If the system later calls tools, the result of a tool call should be treated as a signal.

Example:

```text
Tool action:
Run approved read-only command.

Tool result signal:
Command output, exit code, error output.
```

The tool result is not automatically truth.

It must be checked for:

```text
source
command identity
permission
freshness
exit status
stdout/stderr
expected result
prediction error
```

Tool use must remain downstream of action proposal and safety, not part of uncontrolled signal intake.

---

## Human correction as signal

A human correction is an important signal source.

Example:

```text
User says: "No, that is not what I meant."
```

This should be treated as strong context, but still represented as a signal with uncertainty and update logic.

A human correction may justify model update when:

```text
- the correction is clear
- the source is authorized
- the correction target is known
- the update scope is bounded
```

The system should preserve:

```text
previous model
previous prediction
human correction signal
prediction error
model update
remaining uncertainty
```

---

## Signal source admission

Before a signal source can be used, it should be admitted.

Admission means the system has a record of:

```text
source name
source type
allowed scope
read/write capability
privacy risk
permission status
activation status
expected signal types
safety constraints
revocation method
```

A signal source should not exist as an invisible capability.

If the system can observe something, that observation pathway should be named and governed.

---

## Signal source states

A signal source may have states such as:

```text
available
unavailable
disabled
enabled
permission_required
permission_denied
read_only
read_write
experimental
deprecated
blocked_by_safety
```

The system should represent these states explicitly.

This helps the world model know what can be observed.

---

## Signal source boundaries

Each signal source should define boundaries.

Examples:

```text
ManualTextSignalSource:
- user-provided text only
- no hidden context
- no external reading

SystemStatusSignalSource:
- basic runtime/system facts only
- no secrets
- no full machine scan

ProjectDirectorySignalSource:
- project root only
- read-only
- no file mutation

CameraSignalSource:
- disabled until explicit permission
- no background capture

MicrophoneSignalSource:
- disabled until explicit permission
- no background listening
```

Boundaries should be part of the signal source definition, not informal notes.

## Observed features

Signals should not always be treated as opaque payloads.

The system should eventually extract observed features.

Examples:

Manual text observed features:

```text
non_empty_text
contains_question
contains_correction
contains_restart_instruction
contains_uncertainty_marker
```

System status observed features:

```text
runtime_available
project_path_known
trace_dir_available
python_version_known
```

File state observed features:

```text
file_exists
file_missing
file_changed
folder_present
```

Resource load observed features:

```text
cpu_normal
memory_normal
disk_available
runtime_degraded
```

Predictions should be compared against observed features, not just raw payload text.

---

## Prediction depends on sensorium

The system’s predictions should depend partly on which signal sources are available.

Example:

```text
If only ManualTextSignalSource is enabled,
the system should not predict camera observations.
```

Another example:

```text
If SystemStatusSignalSource is enabled,
the system may predict that runtime status should be observable.
```

The world model should know the current sensorium.

The prediction layer should not expect signals from sources that are disabled or outside boundary.

---

## Signal-source agnostic does not mean unrestricted

The architecture should be signal-source agnostic in design.

That means it can represent many signal types.

It does not mean all signal sources are active or allowed.

Correct distinction:

```text
Architecturally representable: many sources
Currently admitted: only safe bounded sources
Currently enabled: minimal sources for the current phase
```

The first corrected restart should be signal-source agnostic but capability-minimal.

## Example: manual text as one source

World model:

```text
current_project_phase = "foundation restart"
available_signal_sources = ["manual_text", "system_status"]
expected_next_signal_type = "architecture clarification"
```

Prediction:

```text
The next relevant manual text signal may clarify the restart direction.
System status should remain available.
```

Signal source selected:

```text
ManualTextSignalSource
```

Signal:

```text
"No, start from scratch."
```

Prediction comparison:

```text
manual text signal matches expected architecture clarification / restart direction
```

Result:

```text
ModelUpdate: project_phase = "restart from corrected foundation"
```

Manual text was used, but it was not the root of the loop.

The prediction came first.

## Example: system status signal

World model:

```text
runtime expected to be active
trace directory expected to exist
system status source enabled
```

Prediction:

```text
SystemStatusSignalSource should report runtime_available and trace_dir_available.
```

Signal:

```text
runtime_available = true
trace_dir_available = false
```

Prediction error:

```text
partial mismatch: runtime is available, trace directory missing
```

Evidence quality:

```text
medium/high if source is reliable
```

Action:

```text
No autonomous mutation.
Propose creating trace directory or ask for permission.
```

This example shows how non-text signals can drive model update or action proposal.

## Example: memory trace as signal

World model:

```text
current project phase uncertain
memory source enabled
```

Prediction:

```text
Prior traces may contain evidence about whether v0.2 was closed out.
```

Signal:


```text
match: memory source contains relevant v0.2 closeout evidence
```

Evidence quality:

```text
medium, because retrieved memory is context not truth
```

```text
Maybe update project history, but do not treat memory alone as current truth.
```

## Bad signal handling pattern

Bad pattern:

```text
Input arrives.
System assumes it is true.
System updates belief.
System acts.
```

This is unsafe.

It collapses signal into truth and action.

---

## Correct signal handling pattern

Correct pattern:

```text
Prediction exists first.
Signal arrives from a bounded source.
Signal is captured with source metadata.
Signal is compared against prediction.
Prediction error is computed.
Evidence quality is evaluated.
System updates, abstains, gathers more signal, or proposes bounded action.
Trace is written.
```

This is the intended signal discipline.

---

## Signals and uncertainty

Every signal can carry uncertainty.

Uncertainty may come from:

```text
missing payload
ambiguous content
untrusted source
stale observation
partial observation
unexpected source
permission uncertainty
incomplete context
conflicting memory
failed capture
```

The system should preserve uncertainty rather than hiding it.

Uncertainty should influence evidence quality and action readiness.

---

## Signals and privacy

Signal sources can expose sensitive information.

The system should follow minimization:

```text
observe only what is needed
prefer coarse observations over detailed capture
avoid hidden background observation
require permission for sensitive sources
record when sensitive sources are used
allow revocation
```

High-sensitivity sources include:

```text
camera
microphone
active window state
keyboard/mouse activity
broad filesystem access
network content
browser content
private messages
credentials or secrets
```

These should not be part of the early restart.

---

## Signals and action separation

Observation and action must remain separate.

A signal source observes.

An action changes something.

For example:

```text
Filesystem signal source:
observes file state

Filesystem action:
creates, edits, moves, or deletes a file
```

The first may be allowed read-only.

The second requires safety checks, permission, and outcome observation.

Do not blur these categories.

---

## Signals and LLMs

If an LLM is used later, its output should be treated as a signal or proposal.

It should not be treated as truth.

Possible LLM-generated signals:

```text
interpretation proposal
summary proposal
classification proposal
prediction proposal
ambiguity flag
```

These must still be checked against:

```text
world model
prediction
source records
evidence quality
safety rules
```

An LLM should not become the owner of the world model or the authority for action.

---

## Sensorium and world model

The world model should include the current sensorium.

Example fields:

```text
available_signal_sources
enabled_signal_sources
disabled_signal_sources
permission_required_sources
trusted_sources
experimental_sources
source_boundaries
```

This allows the system to generate realistic predictions.

A system cannot predict from sources it does not know it has.

A system should not request signals from sources outside its admitted boundary.

---

## Sensorium and metacognition

The system should know when its sensorium is limited.

Examples:

```text
No camera source available.
No microphone source available.
No file watcher available.
Only manual text is enabled.
System status source failed.
Memory source unavailable.
```

These limitations should affect uncertainty.

A system with limited sensorium should be more cautious.

It should avoid pretending it observed what it cannot observe.



## Design principle summary

The signal system should obey these principles:

```text
1. Signals are observations, not truth.
2. Manual text is one signal source, not the root of cognition.
3. Every signal has a source.
4. Every source has a boundary.
5. Every sensitive source requires explicit permission.
6. Prediction should happen before signal interpretation.
7. Signals should be compared against predictions.
8. Prediction error should drive update pressure.
9. Evidence quality should decide whether update is justified.
10. Observation must remain separate from action.
11. Memory is context, not truth.
12. LLM output is proposal, not authority.
13. Sensorium limitations should increase uncertainty.
14. Capability expansion must be staged and reversible.
```

---

## Final summary

The Bounded Predictive Machine needs a real sensorium, not just a text input box.

But it should not gain broad access all at once.

The corrected architecture should represent many possible signal sources while enabling only safe, bounded sources at first.

The first restart should prove this:

```text
body/system state + priors + world model
→ prediction
→ bounded signal source
→ signal record
→ prediction comparison
→ prediction error
→ evidence quality
→ update/no-update
```

Manual text remains useful.

But it is only one signal source.

The system’s cognition must not begin there.
