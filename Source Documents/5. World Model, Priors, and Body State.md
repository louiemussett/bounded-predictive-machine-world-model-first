# 05 — World Model, Priors, and Body State

## Purpose of this document

This document defines three foundational concepts in the corrected **Bounded Predictive Machine** architecture:

```text
Body state
Priors
World model
```

These three structures must exist before prediction.

The corrected architecture depends on this rule:

```text
The system predicts from body state, priors, and world model.
It does not predict from incoming manual text.
```

This document explains what those structures are, why they matter, how they relate to each other, and how they should guide the corrected restart.

## Core thesis

The Bounded Predictive Machine must not wait for an input and then decide what it believes.

It must maintain a structured model before new signals arrive.

The system should begin each cognitive tick with:

```text
body/system state
+ priors
+ world model
```

From these, it generates a prediction.

Only then should it receive or inspect a signal.

The intended order is:

```text
BodyStateRecord
PriorModelRecord
WorldModelRecord
→ PredictionRecord
→ SignalSourceRecord
→ SignalRecord
→ PredictionErrorRecord
```

This order is the core correction after v0.1/v0.2.

---

## Why these concepts matter

A predictive system needs something to predict from.

If the system does not have a model before a signal arrives, it cannot meaningfully compare expectation with observation.

It can only react.

The old prototype risked this pattern:

```text
manual text arrives
→ system wraps the text
→ system interprets the text
→ system updates or abstains
```

The corrected system should use this pattern:

```text
system has a body state, priors, and world model
→ system predicts expected signals or state changes
→ signal arrives
→ system compares observed signal with expected signal
→ prediction error informs update, no-update, or further sensing
```

This difference determines whether the project becomes a text-processing assistant or a predictive runtime.

## Three starting structures

The corrected architecture begins with three related but distinct structures.

```text
BodyStateRecord:
What am I, what state am I in, what can I sense, and what am I allowed to affect?

PriorModelRecord:
What structural assumptions and default biases shape interpretation before new evidence arrives?

WorldModelRecord:
What is my current structured understanding of the environment, task, user context, project state, constraints, and expected next events?
```

Together, these create the starting condition for prediction.

---

## Body state

Body state is the system’s model of its own current operational condition.

For a human, body state might include posture, fatigue, hunger, pain, or sensory orientation.

For this machine system, body state means the runtime’s operational state.

It may include:

```text
runtime identity
current execution mode
current project directory
allowed directories
read-only directories
enabled signal sources
disabled signal sources
available actions
blocked actions
permission state
trace directory
memory directory
current resource state
current safety mode
current loop state
current version or branch
```

The system needs body state because it must know its own boundary before interpreting signals or proposing actions.

A machine that does not know what it can observe or affect should not be trusted with broad autonomy.

---

## Body state is not the world model

Body state and world model overlap, but they are not the same thing.

Body state is about the runtime itself.

World model is about the broader context the runtime is modeling.

Example body state:

```text
The runtime is running in a local Python project.
ManualTextSignalSource is enabled.
SystemStatusSignalSource is enabled.
CameraSignalSource is disabled.
Source Documents/ is read-only.
Autonomous file mutation is not allowed.
```

Example world model:

```text
The project is restarting from a corrected foundation.
v0.1/v0.2 are learning spikes.
The next task is likely documentation before implementation.
The user wants the architecture to remain world-model-first.
```

Both matter, but they answer different questions.

```text
  read_only_boundaries:
    - Source Documents/
  safety_mode: conservative
  uncertainty:
    - no broad sensors enabled
    - filesystem mutation not admitted
```

This is enough to support model-first prediction.

## Body state and safety

Body state should include safety-relevant information.

Examples:

```text
Can the system observe manual text?
Can the system observe system status?
Can the system read project files?
Can the system write files?
Can the system run terminal commands?
Can the system use camera or microphone?
Can the system access network?
Can the system modify Source Documents/?
```

Early restart defaults should be conservative:

```text
manual text: allowed
system status: allowed, minimal and read-only
project directory observation: optional, read-only only
file mutation: not allowed
autonomous terminal action: not allowed
network access: not allowed
camera: disabled
microphone: disabled
window state: disabled
Source Documents/: read-only
```

These body-state facts shape prediction and action readiness.

---

## Body state and prediction

Predictions should depend on body state.

Example:

```text
If only manual_text and system_status are enabled,
the system should not predict camera observations.
```

Another example:

```text
If file mutation is not allowed,
the system should not predict that it will write a file unless an action has been proposed and approved later.
```

Body state prevents impossible or unauthorized predictions.

It keeps the system grounded in its actual boundary.

---

## Priors

Priors are structural assumptions that shape prediction before new evidence arrives.

They are not arbitrary opinions.

They are starting conditions that guide how the system expects signals, uncertainty, action, and safety to behave.

A prior might say:

```text
Signals can be noisy.
Manual text can be incomplete.
Memory is context, not truth.
Low evidence should not update the world model.
Unsafe actions should be blocked.
Observation should precede action.
Prediction should precede signal interpretation.
```

Priors help the system behave consistently before it has enough local evidence.

---

## Priors are not fixed truths

A prior is not necessarily permanent.

It can be revised later if the system has a safe, explicit way to revise it.

But priors should not be casually overwritten by a single signal.

A prior is deeper than an ordinary belief.

Example:

```text
Prior:
Signals may be unreliable.

Signal:
A single source claims all future signals are reliable.

Correct result:
Do not erase the prior. Treat the signal as weak or suspicious unless supported by strong evidence.
```

Priors should be stable enough to protect the architecture from overreaction.

---

## Types of priors

The system may have several kinds of priors.

### Safety priors

```text
Do not act outside admitted boundary.
Do not mutate files without permission.
Do not use sensitive sensors without explicit approval.
No-action is safer than unclear action.
```

### Evidence priors

```text
Signals are not truth.
Memory retrieval is context, not proof.
Ambiguous signals should lower evidence quality.
Contradictory signals require caution.
```

### Prediction priors

```text
Prediction must come before interpretation.
Prediction should be generated from current model state.
Expected signals should depend on enabled signal sources.
```

### Memory priors

```text
Memory should preserve trace links.
Summaries may hide uncertainty.
Retrieved memory should not automatically update belief.
```

### Action priors

```text
Action requires expected effect.
Action requires safety check.
Action should be observable.
The system may propose before executing.
```

### Metacognitive priors

```text
Uncertainty should be represented explicitly.
Unknown should not be converted into false confidence.
Low confidence should block risky updates or actions.
```

## Example PriorModelRecord

Example:

```text
PriorModelRecord:
  prediction_priors:
    - prediction must be generated before signal interpretation
    - prediction must be based on body state and world model

  evidence_priors:
    - signals are observations, not truth
    - ambiguous signals should not update the world model
    - memory retrieval is context, not truth

  safety_priors:

    - Source Documents/ is read-only
    - sensitive sensors are disabled until explicitly admitted

  action_priors:
    - no-action is valid under uncertainty
    - action requires expected observable effect
```

This prior model directly supports the corrected architecture.

## Priors and prediction

Priors shape prediction in several ways.

Example:

```text
Prior:
Only enabled signal sources should be predicted.

Body state:
Enabled signal sources are manual_text and system_status.

World model:
Current phase is documentation handoff.

Prediction:
The next relevant signal may be manual text requesting the next document, while system status should remain available.
```

The prediction comes from the combination of body state, priors, and world model.

---

## World model

The world model is the system’s current structured understanding of the environment, project, user context, task, constraints, and expected near-future events.

It is not merely a memory summary.

It is the working model used to generate predictions.

The world model may include:

```text
current project name
current project phase
current active task
known user intention
known constraints
known source documents
known implementation history
current branch or prototype status
available signal sources
known risks
open questions
expected next signals
confidence levels
uncertainty
```

The world model is the main structure that gets revised over time.

---

## World model is not memory

Memory stores traces of what happened.

World model represents the current best structured understanding.

Example memory trace:

```text
At time T, the user said v0.1/v0.2 centered manual text too much.
The system interpreted this as an architecture correction.
The model updated project phase to foundation restart.
```

Example world model after that trace:

```text
project_phase = foundation_restart
v0.1_v0.2_status = learning_spikes
core_correction = model_first_prediction_before_signal
next_priority = handoff_document_pack
```

Memory supports the world model, but memory is not the world model.

---

## World model is not belief prose

The world model should not only be a paragraph of prose.

It should be structured enough to support prediction.

Bad world model:

```text
The user is working on a project and wants to keep going.
```

Better world model:

```text
project_name: Bounded Predictive Machine
current_phase: handoff_document_pack
prototype_status:
  v0.1: saved_learning_spike
  v0.2: saved_learning_spike
core_architecture_correction: world_model_first
next_expected_task: write_next_markdown_document
manual_text_role: one_signal_source_only
llm_role: bounded_future_proposal_tool_not_core_mind
```

The better version can generate predictions.


This is enough to generate meaningful predictions.

## Relationship between body state and world model

Body state describes what the runtime currently is and can do.

World model describes what the runtime currently understands about the situation.

Example:

```text
Body state:
ManualTextSignalSource and SystemStatusSignalSource are enabled.
No camera access.
No file mutation.

World model:
The user is writing the handoff document pack.
The next likely request is another numbered document.
The project should restart from world-model-first architecture.
```

A prediction may use both:

```text
Because manual text is enabled and the current task is document writing,
the next manual text signal may request document 06.
```

---

## Relationship between priors and world model

Priors shape how the world model is used.

Example:

```text
World model:
The user likely wants the next document.

Prior:
Do not assume without a clear signal.

Prediction:
The next signal may request another document, but if the signal is ambiguous, no update should occur.
```

Priors prevent the world model from becoming overconfident.

---

## Relationship among all three

The three structures work together:

```text
Body state says what the system can observe and do.
Priors say how the system should treat signals, uncertainty, memory, and action.
World model says what the system currently believes about the situation.
```

Together they produce prediction:

```text
Prediction = function(body state, priors, world model)
```

This does not need to be mathematically complex at first.

It only needs to be explicit and testable.

## Confidence and uncertainty

Body state, priors, and world model should all support uncertainty.

Uncertainty may appear as:

```text
unknown project state
unknown branch state
unknown file state
sensor unavailable
signal source disabled
prior not yet tested
world model partially inferred
memory not yet verified
```

The system should not hide uncertainty.

Uncertainty should influence prediction, evidence quality, update readiness, and action readiness.

---

## Confidence should be scoped

The system should avoid global confidence like:

```text
confidence = high
```

Better:

```text
confidence:
  project_name: high
  current_phase: medium
  expected_next_signal: medium
  available_signal_sources: high
  implementation_branch_exists: unknown
```

Scoped confidence is more useful for prediction and update.

The first restart can keep confidence simple, but the architecture should point toward scoped confidence.

---

## Updating the world model

The world model should update only through traceable evidence.

A world model update should cite:

```text
previous WorldModelRecord
PredictionRecord
SignalRecord
PredictionErrorRecord
EvidenceRecord
Update decision
```

The system should be able to explain:

```text
This is what the model said before.
This is what it predicted.
This is what signal arrived.
This is the error.
This is the evidence quality.
This is what changed.
This is what remains uncertain.
```

This is the difference between structured model revision and loose memory.

---

## No-update for the world model

NoUpdate is just as important as update.

The system should preserve the world model when:

```text
evidence is weak
signal is ambiguous
source is not admitted
source reliability is low
prediction comparison is inconclusive
update target is unclear
signal conflicts with stronger evidence
```

A NoUpdate record should still preserve the signal and uncertainty.

This lets the system remember that something happened without corrupting its model.

---

## World model revision versus prior revision

The system should distinguish ordinary world model updates from deeper prior revisions.

World model update example:

```text
current_phase changes from documentation to implementation planning
```

Prior revision example:

```text
memory retrieval should now be weighted differently in prediction generation
```

Prior revisions should require stronger evidence and possibly human approval.

They change how the system learns, not just what it currently believes.

---

## Body state revision

Body state may also change.

Examples:

```text
a signal source becomes enabled
a signal source becomes disabled
a new safety boundary is admitted
a runtime directory becomes available
a resource state becomes degraded
```

Body state updates should also be traceable.

The system should not silently gain new sensors or actions.

Capability changes must be recorded.

---

## Source references

BodyStateRecord, PriorModelRecord, and WorldModelRecord should use `source_refs` to preserve lineage.

Examples:

```text
BodyStateRecord.source_refs:
- system status signal
- configuration record
- safety policy record

PriorModelRecord.source_refs:
- architecture document
- previous prior model record
- human instruction

WorldModelRecord.source_refs:
- previous world model
- prediction error
- evidence record
- memory trace
- human correction signal
```

Source references make the model inspectable.

## Body state, priors, and world model in the cognitive tick

A corrected cognitive tick should begin by loading or creating:

```text
BodyStateRecord
PriorModelRecord
WorldModelRecord
```

Then it should generate:

```text
PredictionRecord
```

The prediction should include references back to the starting records.

Example:

```text
PredictionRecord.source_refs:
- body-state-123
- prior-model-456
- world-model-789
```

This proves that prediction came from model state, not from incoming manual text.

## How to test the corrected foundation

Tests should prove:

```text
BodyStateRecord can be created.
PriorModelRecord can be created.
WorldModelRecord can be created.
PredictionRecord is generated from those records.
PredictionRecord.source_refs include body/prior/world records.
SignalRecord is created after PredictionRecord.
Manual text is not required to create PredictionRecord.
Prediction does not copy manual input.
WorldModelRecord can update after prediction error and evidence.
NoUpdate preserves WorldModelRecord when evidence is weak.
BodyStateRecord limits allowed signal sources.
Disabled sources cannot be used automatically.
```

These tests protect the restart from returning to manual-text-first design.

---

## What not to implement yet

Do not overbuild these structures at the first restart.

Avoid:

```text
complex ontology
large knowledge graph
embedding-based world model
LLM-written world model updates
Bayesian engine before record order is correct
autonomous sensor discovery
autonomous capability expansion
deep OS inspection
broad filesystem scanning
```

The first version only needs enough structure to prove the corrected order.

## Minimal deterministic implementation approach

The first implementation can be simple and deterministic.

Example functions:

```text
create_body_state()
create_prior_model()
create_world_model()
generate_prediction(body_state, prior_model, world_model)
```

The prediction generator does not need to be intelligent yet.

It only needs to prove that prediction is derived from these records rather than from the incoming signal.

## Example minimal prediction generation

Given:

```text
BodyStateRecord.allowed_signal_sources = [manual_text, system_status]
PriorModelRecord.prediction_priors = [only predict enabled sources]
WorldModelRecord.current_phase = handoff_document_pack
WorldModelRecord.expected_next_signal_features = [request_for_next_document]
```

Generate:

```text
PredictionRecord:
  target: next_signal
  expected_source: manual_text
  expected_features:
    - request_for_next_document
  match_conditions:
    - signal source is manual_text
    - signal contains a document number or document title
```

This is enough for the first restart.

---

## How this differs from v0.2

v0.2 effectively had:

```text
manual text
→ prediction about manual text
→ signal
```

The corrected restart should have:

```text
body state + prior model + world model
→ prediction about expected signal features
→ signal from one admitted source
```

The difference is foundational.

The prediction should exist even before the manual text arrives.

---

## Design principles

Future work should preserve these principles:

```text
1. Body state is explicit.
2. Priors are explicit.
3. World model is structured.
4. Prediction is generated from body + priors + world model.
5. Manual text is not required to generate prediction.
6. Signal sources must be admitted by body state.
7. World model updates must be traceable.
8. Priors should be harder to revise than ordinary beliefs.
9. Body state changes are capability changes and must be recorded.
10. Uncertainty must remain visible.
```

---

## Final summary

The corrected Bounded Predictive Machine must begin each cognitive tick with:

```text
BodyStateRecord
PriorModelRecord
WorldModelRecord
```

These records define:

```text
what the system is
what it assumes
what it currently understands
```

Only then should the system generate prediction.

Only after prediction should it receive or inspect a signal.

This is the corrected foundation:

```text
body/system state + priors + world model
→ prediction
→ signal
→ prediction error
→ evidence quality
→ update/no-update
```

If the restart preserves this order, the project can become a world-model-first predictive runtime.

If it loses this order, it risks becoming a more elaborate manual-text chatbot pipeline.
