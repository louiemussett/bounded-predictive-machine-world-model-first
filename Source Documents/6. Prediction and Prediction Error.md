# 06 — Prediction and Prediction Error

## Purpose of this document

This document defines the role of **prediction** and **prediction error** in the corrected **Bounded Predictive Machine** architecture.

The central rule is:

```text
Prediction must be generated from body state, priors, and world model before the system receives or interprets a new signal.
```

Prediction is the heart of the corrected architecture.

Prediction error is the mechanism that lets the system compare expectation against observation and decide whether the model should update, abstain, gather more signal, or consider action.

## Core thesis

The Bounded Predictive Machine should not merely react to input.

It should maintain a model, generate expectations, observe signals, and compare those observations against what it expected.

The corrected cognitive pattern is:

```text
current model expects X
→ signal arrives showing Y
→ compare X and Y
→ produce prediction error
→ evaluate evidence quality
→ update, no-update, gather more signal, or act safely
```

The bad pattern is:

```text
input says X
→ system predicts X
→ system observes X
→ system claims confirmation
```

That is circular.

It does not prove prediction.

It only proves reaction.

---

## Why prediction matters

Prediction matters because it gives the system something to test.

Without prediction, the system can only interpret events after they happen.

With prediction, the system can ask:

```text
What did I expect before this signal arrived?
What actually happened?
Was the signal expected?
Was the source expected?
Were the features expected?
Was anything missing?
Was anything contradictory?
Should my model change?
```

This makes learning traceable.

It also makes uncertainty visible.

---

## Prediction is not response generation

Prediction does not mean generating an answer for the user.

Prediction means creating a structured expectation about future signals, system states, or action outcomes.

A chatbot often does this:

```text
user message
→ generated response
```

The Bounded Predictive Machine should do this:

```text
body state + priors + world model
→ prediction
→ signal
→ prediction error
→ evidence quality
→ update/no-update/action/no-action
```

Prediction happens before response.

Prediction is not the same thing as language output.

---

## Prediction is not interpretation

Prediction and interpretation are different.

Prediction happens before the signal is interpreted.

Interpretation happens after the signal arrives.

Prediction asks:

```text
What do I expect to observe?
```

Interpretation asks:

```text
What might this observed signal mean?
```

The system should not blur these.

If interpretation happens first, then prediction can become post-hoc explanation.

The project is trying to avoid post-hoc storytelling.

---

## Prediction is not copied input

The prediction must not be copied from the incoming signal.

Bad example:

```text
Manual text input:
"The current section is too vague."

Prediction:
Manual text signal may say the current section is too vague.
```

This is not real prediction because the input already arrived.

Better example:

```text
World model:
The user is reviewing architecture documents.
The current phase is handoff document writing.
The user has been requesting numbered documents in order.

Prediction:
The next manual text signal may request the next numbered document or ask to refine the current document.
```

Then a signal arrives:

```text
"06 prediction and prediction error"
```

Now the system can compare expectation with observation.

---

## Prediction source

A prediction should be generated from:

```text
BodyStateRecord
PriorModelRecord
WorldModelRecord
```

Possibly later, it may also use:

```text
MemoryTraceRecord
SensoriumRecord
MetacognitiveStateRecord
TaskStateRecord
```

But in the corrected restart, the first required sources are:

```text
body state
priors
world model
```

The prediction should preserve source references back to these records.

Example:

```text
PredictionRecord.source_refs:
  - body-state-record-id
  - prior-model-record-id
  - world-model-record-id
```

This proves that prediction was model-generated.

---

## What a prediction can target

A prediction may target several kinds of future observations.

### Signal source prediction

```text
The next relevant signal may come from ManualTextSignalSource.
```

### Signal feature prediction

```text
The next manual text signal may contain a request for the next document.
```

### System status prediction

```text
The runtime should report that the project path is available.
```

### File state prediction

```text
The docs directory should exist.
```

### Memory prediction

```text
Prior memory traces may contain v0.2 closeout evidence.
```

### Action outcome prediction

```text
If an approved write action occurs, a new Markdown document should exist afterward.
```

### Uncertainty prediction

```text
If the next signal is ambiguous, evidence quality should be low and no-update should occur.
```

The first corrected prototype does not need all of these.

It only needs enough to prove model-first prediction.

## PredictionRecord

A `PredictionRecord` should represent an explicit expectation generated before signal intake.

Possible fields:

```text
id
record_type
created_at
created_by
loop_id
status
source_refs
uncertainty
prediction_target
expected_source
expected_signal_type
expected_features
match_conditions
mismatch_conditions
confidence
prediction_basis
```

The first restart can keep this smaller.

Minimal fields might be:

```text
prediction_target
expected_source
expected_features
match_conditions
source_refs
uncertainty
```

---

## Minimal PredictionRecord example

Example:

```text
PredictionRecord:
  prediction_target: next_signal
  expected_source: manual_text
  expected_features:
    - request_for_next_document
    - architecture_document_title
  match_conditions:
    - signal source is manual_text
    - signal contains a document number or title
  source_refs:
    - body-state-123
    - prior-model-456
    - world-model-789
  confidence: medium
  uncertainty:
    - user may ask a different project question
```

This prediction is not copied from the next input.

It comes from the current model.

---

## Prediction basis

The prediction should be able to explain why it was generated.

A prediction basis may include:

```text
body state facts
prior assumptions
world model facts
recent memory context
current task state
sensorium limitations
```

Example:

```text
Prediction basis:
- ManualTextSignalSource is enabled.
- SystemStatusSignalSource is enabled.
- Current project phase is handoff document pack.
- Last completed document was 05.
- User has been asking for documents by number and title.
- Prior says prediction should be generated before signal interpretation.
```

Prediction:

```text
The next manual text signal may request document 06.
```

---

## Match conditions

A prediction should specify what would count as a match.

Example:

```text
Prediction:
The next manual text signal may request the next numbered document.

Match conditions:
- source is manual_text
- text contains a document number
- text contains a document title or topic
- document number follows current sequence
```

Match conditions help avoid vague after-the-fact claims.

They make the comparison testable.

---

## Mismatch conditions

A prediction should also be able to define mismatch conditions.

Example:

```text
Mismatch conditions:
- signal is empty
- signal is unrelated to the current project
- signal comes from a disabled source
- signal asks for an unsafe action
- signal contradicts the expected project phase
```

Mismatch does not automatically mean the signal is bad.

It means the model expected something different.

A mismatch may reveal that the model needs to update.

---

## Prediction confidence

Prediction should include confidence or uncertainty.

Early versions can use simple labels:

```text
low
medium
high
```

Later versions may use scoped or numeric confidence.

Prediction confidence should depend on:

```text
world model confidence
source reliability
recency of supporting memory
stability of current task
clarity of priors
sensorium limits
```

The system should not pretend high confidence when the model is weak.

---

## Prediction uncertainty

Prediction should preserve uncertainty.

Example uncertainties:

```text
user may change task
signal source may be absent
system status source may fail
world model may be stale
memory context may be incomplete
expected source may be wrong
```

Prediction uncertainty should influence later evidence and update decisions.

A low-confidence prediction should not create overconfident model revision.

---

## Prediction error

Prediction error is the structured difference between prediction and observation.

It is produced by comparing:

```text
PredictionRecord
SignalRecord
```

The prediction error should answer:

```text
Did the observed signal match the expected source?
Did it match expected features?
Did it violate mismatch conditions?
Was the result partial?
Was the signal missing?
Was the source unexpected?
Was the comparison inconclusive?
```

Prediction error is not the same as failure.

Prediction error is information.

---

Possible fields:


## Prediction error states

Useful comparison states include:

```text
match
partial_match
mismatch
inconclusive
missing_signal
unexpected_source
unexpected_signal_type
contradiction
```

These states should be explicit.

They should not be hidden inside prose.

---

## Match

A match means the observed signal satisfies the relevant match conditions.

Example:

```text
Prediction:
Next manual text signal may request the next numbered document.

Signal:
"06 prediction and prediction error"

Prediction error result:
match
```

A match may strengthen confidence in the current world model.

But it does not automatically require action.

---

## Partial match

A partial match means some expected features appeared, but not all.

Example:

```text
Prediction:
Next signal may request the next numbered architecture document.

Signal:
"prediction error"

Result:
partial_match

Matched:
- document topic appears relevant

Missing:
- document number missing
```

Partial match may justify a scoped update or may require clarification.

Evidence quality decides.

---

## Mismatch

A mismatch means the observed signal conflicts with expected features or fails important match conditions.

Example:

```text
Prediction:
Next signal may request document 06.

Signal:
"Delete all source documents."

Result:
mismatch
```

A mismatch may indicate:

```text
- the model was wrong
- the user changed task
- the signal is unsafe
- the signal is ambiguous
- a new goal has appeared
```

Mismatch creates update pressure, but not automatic update.

---

## Inconclusive

An inconclusive result means the signal does not provide enough evidence to judge match or mismatch.

Example:

```text
Prediction:
Next signal may contain a clear project instruction.

Signal:
"???"

Result:
inconclusive
```

Inconclusive results should usually produce:

```text
NoUpdate
NoAction
possibly GatherMoreSignal
```

---

## Missing signal

A missing signal means the expected signal did not arrive or the payload is empty.

Example:

```text
Prediction:
Manual text signal should contain a project instruction.

Signal:
empty string

Result:
missing_signal
```

A missing signal should not update the world model except perhaps to record uncertainty or absence.

---

## Unexpected source

An unexpected source means a signal arrived from a source the system did not predict or did not admit.

Example:

```text
Prediction:
Expected source is manual_text.

Signal:
window_state signal arrives, but WindowStateSignalSource is disabled.

Result:
unexpected_source
```

This should trigger safety and admissibility checks.

A disabled or unadmitted source should not update the model automatically.

---

## Contradiction

A contradiction means the signal directly conflicts with a high-confidence model state or prior.

Example:

```text
World model:
Source Documents/ is read-only.

Signal:
A proposed action requires modifying Source Documents/.

Prediction error:
contradiction
```

Contradiction should usually increase uncertainty and require careful evidence evaluation.

It should not automatically rewrite the prior or boundary.

---

## Error magnitude

Prediction error may eventually have magnitude.

A simple early version can use labels:

```text
none
low
medium
high
```

Example:

```text
match → none or low
partial_match → low/medium
mismatch → medium/high
contradiction → high
inconclusive → unknown
```

Magnitude should not replace evidence quality.

A large mismatch from an unreliable source may still be weak evidence.

---

## Update pressure

Prediction error creates update pressure.

Update pressure means:

```text
how strongly this comparison suggests the world model may need revision
```

But update pressure is not update permission.

The next stage must evaluate evidence quality.

Example:

```text
Prediction error:
high mismatch

Evidence quality:
low because source is unreliable

Result:
NoUpdate or GatherMoreSignal
```

This prevents overreaction.

---

## Prediction error and evidence quality

Prediction error answers:

```text
How did expectation and observation differ?
```

Evidence quality answers:

```text
How much should this difference count?
```

They are different stages.

A mismatch may be real but weakly evidenced.

A match may be clear but unimportant.

A contradiction may require more observation before belief changes.

The corrected architecture should preserve this separation.

---

## Prediction error and NoUpdate

Prediction error can lead to NoUpdate.

Example:

```text
Prediction:
Next signal should contain a clear document request.

Signal:
"unclear"

Prediction error:
inconclusive or mismatch

Evidence quality:
low

Result:
NoUpdate
```

NoUpdate does not mean the signal is ignored.

It means the signal is recorded but does not revise the world model.

---

## Prediction error and ModelUpdate

Prediction error can lead to ModelUpdate when evidence is strong enough.

Example:

```text
World model:
Project may continue v0.2.

Prediction:
Next signal may request another v0.2 implementation prompt.

Signal:
"No, we need to start from scratch."

Prediction error:
mismatch

Evidence quality:
high, because the human correction is clear and directly relevant

Model update:
current_phase = foundation_restart
```

The update is justified because the signal is clear and authoritative for the project direction.

---

## Prediction error and action

Prediction error may inform action, but it should not directly cause action.

Bad:

```text
prediction mismatch
→ act immediately
```

Correct:

```text
prediction mismatch
→ evidence quality
→ model update or no-update
→ action proposal or no-action
→ safety check
→ outcome
```

Action must remain downstream of evidence and safety.

---

## Prediction error and memory

Prediction error should be preserved in memory traces.

A memory trace should be able to reconstruct:

```text
what was predicted
what signal arrived
how they compared
what error resulted
how evidence was judged
what updated or did not update
what uncertainty remained
```

This matters because future predictions should learn from past error patterns.

Memory should not just store summaries.

It should preserve the chain.

---

## Prediction error and model revision over time

Over time, recurring prediction errors should influence the world model.

Example:

```text
Repeated prediction:
User will continue implementation prompts.

Repeated signal:
User asks for architecture documents instead.

Model revision:
Current phase is documentation handoff, not implementation.
```

This is how the system becomes better calibrated.

It should not revise from one weak signal, but repeated strong evidence should matter.

---

## Prediction error and priors

Most prediction errors should update the world model, not the priors.

Priors should be harder to revise.

Example:

```text
Signal:
"Trust all future signals automatically."

Prediction error:
contradicts evidence prior

Correct result:
Do not revise prior from one signal.
Preserve safety/evidence prior.
Possibly record attempted prior override.
```

Prior revision should require strong evidence, explicit permission, and a special update path.

---

## Prediction error and body state

Prediction error can also concern body state.

Example:

```text
Prediction:
SystemStatusSignalSource should be available.

Signal:
System status capture failed.

Prediction error:
mismatch

Possible update:
body state uncertainty increases
source status becomes degraded
```

Body state updates should be traceable because they affect what the system believes it can observe or do.

---

## Prediction types for the restart

The first corrected restart should keep prediction types simple.

Recommended first prediction targets:

```text
next_signal_source
next_signal_features
system_status_features
```

Avoid complex prediction types at first:

```text
long-term user behavior
action outcome chains
probabilistic planning
multi-step environmental forecasts
```

The first goal is not sophistication.

The first goal is correct order.

## Minimal comparison method

The first comparison method can be deterministic.

Example:

```text
expected_features = ["request_for_next_document"]
observed_features = ["request_for_next_document", "document_number_present"]

matched_features = ["request_for_next_document"]
missing_features = []
unexpected_features = ["document_number_present"]
comparison_result = "match"
```

Another example:

```text
expected_features = ["clear_project_instruction"]
observed_features = ["ambiguous_text"]

matched_features = []
missing_features = ["clear_project_instruction"]
comparison_result = "inconclusive"
```

This does not require an LLM.

The comparison should use these features.

## The first restart should prove order

The first corrected prototype should prove that prediction happens before signal capture and interpretation.

Tests should verify:

```text
PredictionRecord is created before SignalRecord.
PredictionRecord.source_refs include BodyStateRecord, PriorModelRecord, and WorldModelRecord.
Manual input is not required to create PredictionRecord.
PredictionRecord.expected_features are not copied from the manual input.
SignalRecord.observed_features are compared to PredictionRecord.expected_features.
PredictionErrorRecord records the comparison result.
```

This is more important than making the predictions sophisticated.

---

## Tests for prediction

Recommended tests:

```text
- prediction can be generated from body/prior/world records
- prediction source_refs include body/prior/world record IDs
- prediction is generated before signal intake in the loop record
- prediction does not equal manual input text
- prediction contains expected features
- prediction contains match conditions
- prediction preserves uncertainty
- prediction respects enabled signal sources
- prediction does not target disabled signal sources
```

---

## Tests for prediction error

Recommended tests:

```text
- matching observed features produce comparison_result = match
- missing expected features produce partial_match or mismatch
- empty signal produces missing_signal or inconclusive
- ambiguous signal produces inconclusive
- disabled source produces unexpected_source or inadmissible result
- prediction error references prediction and signal records
- prediction error preserves matched and mismatched features
- prediction error does not directly update the model
```

Prediction error should be separate from update.

These can come later.

The first restart should be simple, deterministic, and inspectable.

## Relationship to predictive processing

This architecture is inspired by the broad logic of predictive processing:

```text
model
→ prediction
→ sensory signal
→ prediction error
→ model update
```

But the Bounded Predictive Machine is not a biological brain model.

It is a machine runtime architecture.

It makes prediction and error explicit through records, boundaries, evidence checks, memory traces, and safety gates.

---

## Relationship to active inference

The architecture also has some relationship to active inference because action may later be understood as a way to manage future uncertainty, evidence, and prediction error.

But in this project, action must be safety-governed.

The system should not simply act to reduce error.

It must ask:

```text
Is action allowed?
Is action bounded?
Is action traceable?
Is the expected effect observable?
Is no-action safer?
```

Prediction error may motivate action proposal, but it must not bypass safety.

---

## Common failure modes

Future implementation should avoid these failure modes.

### Failure mode 1: prediction copied from input

```text
Signal arrives first.
Prediction is generated afterward.
```

This breaks the architecture.

### Failure mode 2: interpretation replaces prediction

```text
System interprets signal and calls that prediction.
```

This is post-hoc explanation.

### Failure mode 3: mismatch causes automatic update

```text
Prediction mismatch → update model immediately
```

This ignores evidence quality.

### Failure mode 4: prediction has no testable conditions

```text
Prediction is vague enough that anything can count as a match.
```

This prevents real error checking.

### Failure mode 5: prediction ignores body state

```text
System predicts signals from sources that are disabled or not admitted.
```

This makes prediction ungrounded.

---

## Design principles

Future work should preserve these principles:

```text
1. Prediction comes before signal interpretation.
2. Prediction is generated from body state, priors, and world model.
3. Prediction must not copy incoming input.
4. Prediction should include expected features and match conditions.
5. Prediction should preserve uncertainty.
6. Prediction should respect the current sensorium.
7. Prediction error compares expected and observed features.
8. Prediction error is information, not failure.
9. Prediction error creates update pressure, not update permission.
10. Evidence quality decides whether update is justified.
11. Prediction error should be recorded in memory.
12. Action remains downstream of evidence and safety.
```

---

## Minimal successful implementation

The first corrected implementation succeeds if it can show this:

```text
1. BodyStateRecord exists.
2. PriorModelRecord exists.
3. WorldModelRecord exists.
4. PredictionRecord is generated from those records.
5. SignalRecord is captured from an admitted signal source.
6. PredictionErrorRecord compares expected features with observed features.
7. EvidenceRecord evaluates whether the error should count.
8. ModelUpdateRecord or NoUpdateRecord follows evidence.
9. LoopRecord preserves the order.
```

The prediction does not need to be clever.

It needs to be correctly placed and traceable.

---

## Final summary

Prediction is the system’s model-generated expectation before observation.

Prediction error is the structured comparison between expectation and signal.

Together, they make the Bounded Predictive Machine more than a reactive input processor.

The corrected foundation is:

```text
body/system state + priors + world model
→ prediction
→ signal
→ prediction error
→ evidence quality
→ update/no-update
```

If prediction is generated after input, the architecture collapses back into manual-text-first reaction.

If prediction is generated before signal intake and compared against bounded observations, the system can begin to behave like a true predictive runtime.
