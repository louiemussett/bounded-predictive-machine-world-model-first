# 07 — Evidence, Uncertainty, and Abstention

## Purpose of this document

This document defines the role of **evidence**, **uncertainty**, and **abstention** in the corrected **Bounded Predictive Machine** architecture.

The key idea is:

```text
Prediction error creates update pressure.
Evidence quality decides whether that pressure should count.
Uncertainty can block update or action.
Abstention is an intelligent outcome.
```

The system should not update its world model or act merely because a signal arrived or because a prediction mismatch occurred.

It must judge whether the signal is good enough evidence.

When evidence is weak, unclear, unsafe, or outside boundary, the system should be able to say:

```text
I do not know enough.
NoUpdate.
NoAction.
Gather more signal.
```

## Core thesis

A bounded predictive machine should not believe, update, remember as truth, or act simply because something appeared as a signal.

A signal must pass through evidence evaluation.

The corrected sequence is:

```text
prediction
→ signal
→ prediction error
→ evidence quality
→ update / no-update / gather-more-signal
→ action / no-action
```

Evidence quality is the gate between prediction error and model revision.

Uncertainty is not a defect to hide.

Uncertainty is a state the system must represent and use.

Abstention is not failure.

Abstention is the correct behavior when the system does not have enough reliable evidence to update or act.

---

## Why evidence matters

Prediction error tells the system that expectation and observation differ.

But difference alone is not enough.

A mismatch may happen because:

```text
- the world model is wrong
- the signal source is unreliable
- the signal is ambiguous
- the signal is stale
- the observation is incomplete
- the prediction was too vague
- the expected feature was poorly defined
- the signal came from an unadmitted source
- the runtime is degraded
- the user changed task
- the system misunderstood the context
```

The system must therefore ask:

```text
How much should this signal count?
```

That is the role of evidence quality.

---

## Evidence is not the same as signal

A signal is an observation candidate.

Evidence is a signal or set of signals judged in relation to prediction, source, boundary, reliability, and uncertainty.

Bad pattern:

```text
signal arrived
→ update belief
```

Correct pattern:

```text
signal arrived
→ compare with prediction
→ evaluate evidence quality
→ update only if justified
```

The system should treat signals as candidates for evidence, not evidence automatically.

---

## Evidence is not the same as prediction error

Prediction error answers:

```text
How did observation differ from expectation?
```

Evidence quality answers:

```text
How much should that difference matter?
```

Example:

```text
Prediction:
The next signal should contain a clear project instruction.

Signal:
"???"

Prediction error:
inconclusive / mismatch

Evidence quality:
low

Result:
NoUpdate and NoAction
```

The prediction error exists, but the evidence is not strong enough to revise the world model.

---

## Evidence quality

Evidence quality is the system’s judgment of whether a signal and prediction comparison are reliable enough to support model revision or action readiness.

Evidence quality may depend on:

```text
source reliability
signal clarity
signal freshness
signal completeness
admissibility
boundary status
prediction error type
matched features
missing features
unexpected features
uncertainty count
conflicts with current model
conflicts with priors
consistency with memory
repeatability
human authority over the topic
```

The first implementation can use simple deterministic rules.

Later implementations may use more sophisticated scoring.

---

## Evidence quality labels

Early versions can use simple labels:

```text
none
low
medium
high
inconclusive
inadmissible
```

Suggested meanings:

```text
none:
No meaningful signal or no evidence.

low:
Signal exists, but it is ambiguous, weak, incomplete, or unreliable.

medium:
Signal is clear enough for scoped update consideration, but uncertainty remains.

high:
Signal is clear, relevant, admissible, and strongly supports update.

inconclusive:
Signal does not allow a reliable judgment.

inadmissible:
Signal is outside boundary, from a disabled source, or not allowed to affect the model.
```

Labels should be explained by reasons.

The system should not output labels without traceable justification.

---

## Evidence score

v0.2 experimented with transparent scoring.

That pattern should be preserved later, but applied to the corrected architecture.

A simple score may include:

```text
base score from evidence label
prediction error adjustment
source reliability adjustment
uncertainty penalty
boundary/admissibility penalty
```

Example:

```text
score = base_score - prediction_penalty - uncertainty_penalty - boundary_penalty
```

The exact formula can change.

The important principle is transparency.

The system should show why the score was produced.

## Evidence reasons

Every evidence judgment should include reasons.

Examples:

```text
signal is empty
signal meaning is unclear
signal source is admitted
signal source is disabled
prediction result is match
prediction result is inconclusive
observed features match expected features
expected features are missing
source reliability is unknown
human correction is clear
signal conflicts with safety prior
```

Reasons make the evidence gate inspectable.

---

## Source reliability

Evidence quality depends partly on source reliability.

Examples:

```text
manual text from user: often high for user intention, not necessarily high for external facts
system status: high for narrow runtime facts if source is working
memory trace: medium context, not truth
LLM proposal: low/medium proposal, not authority
camera signal: variable, requires perception confidence
microphone transcript: variable, may contain transcription errors
terminal output: high for command result if command was approved and correctly scoped
```

Reliability should be scoped.

A source can be reliable for one thing and unreliable for another.

Example:

```text
The user is authoritative about what they want next.
The user is not automatically authoritative about whether a file exists on disk.
```

---

## Admissibility

A signal may be clear but inadmissible.

A signal is inadmissible when it comes from a source or pathway that is not allowed to affect the system.

Examples:

```text
camera signal arrives while camera source is disabled
window state appears without permission
memory trace is corrupted
terminal output comes from an unapproved command
file state comes from outside project boundary
```

Inadmissible signals should not update the world model except perhaps to record a boundary violation or uncertainty.

Admissibility is not the same as truth.

It is about whether the system is allowed to use the signal.

---

## Boundary status

Evidence quality must consider boundary.

A signal may be:

```text
inside_boundary
outside_boundary
boundary_unknown
permission_required
permission_denied
read_only_observation_allowed
unsafe_source
```

Boundary uncertainty should reduce evidence quality and action readiness.

Signals outside boundary should generally produce:

```text
NoUpdate
NoAction
SafetyReview or HumanReview
```

---

## Uncertainty

Uncertainty is a first-class state.

The system should explicitly represent what it does not know.

Uncertainty may appear in:

```text
BodyStateRecord
PriorModelRecord
WorldModelRecord
PredictionRecord
SignalRecord
PredictionErrorRecord
EvidenceRecord
ModelUpdateRecord
NoUpdateRecord
ActionProposalRecord
NoActionRecord
SafetyCheckRecord
OutcomeRecord
MemoryTraceRecord
LoopRecord
```

Uncertainty should not be hidden in vague prose.

It should be recorded as structured flags, reasons, or unresolved questions.

---

## Sources of uncertainty

Uncertainty can come from many places.

### Signal uncertainty

```text
empty signal
ambiguous text
missing payload
stale observation
noisy sensor
partial capture
unknown source reliability
```

### Prediction uncertainty

```text
weak world model
low confidence prediction
vague match conditions
limited sensorium
unclear expected source
```

### Evidence uncertainty

```text
conflicting signals
low reliability source
insufficient observations
unverified memory
boundary ambiguity
```

### Model uncertainty

```text
unknown project phase
unknown current branch
uncertain task state
incomplete memory
open questions
```

### Action uncertainty

```text
unclear target
unknown side effects
unobservable outcome
missing permission
safety boundary unclear
```

---

## Uncertainty should affect behavior

Uncertainty should not merely be recorded.

It should change what the system is allowed to do.

Examples:

```text
High uncertainty about signal meaning → NoUpdate
High uncertainty about action target → NoAction
High uncertainty about source permission → block signal use
High uncertainty about file boundary → no file action
High uncertainty about outcome observability → do not execute action
```

Uncertainty should push the system toward caution.

---

## Uncertainty gate

v0.2 introduced an uncertainty gate.

That idea should be preserved in the corrected architecture.

An uncertainty gate decides whether the system may continue toward update or action.

It may produce decisions such as:

```text
proceed
abstain
no_update
no_action
gather_more_signal
human_review_required
blocked_by_boundary
```

The gate should consider:

```text
evidence quality
prediction error type
source reliability
uncertainty count
boundary status
action risk
```

---

A future `UncertaintyGateRecord` may include:


The first restart may not need a separate uncertainty gate immediately.

But the concept should remain part of the architecture.

---

## Abstention

Abstention means the system explicitly declines to update, act, or claim certainty because evidence is insufficient or unsafe.

Abstention is not silence.

It is a recordable outcome.

Examples:

```text
not enough evidence to update world model
not enough clarity to propose action
action boundary unclear
signal source not admitted
prediction comparison inconclusive
memory retrieval insufficient
human review required
```

Abstention is one of the most important safety behaviors in the project.

---

## Abstention is intelligent

A system that always updates is not smarter.

A system that always acts is not safer.

A bounded predictive machine should know when not to move.

Abstention is intelligent when:

```text
signal is unclear
evidence is weak
prediction error is inconclusive
source is unreliable
permission is missing
boundary is uncertain
memory is insufficient
action risk is too high
```

The system should preserve the reason for abstention so the uncertainty can be revisited later.

---

Possible fields:

```text

```

## NoUpdate

NoUpdate means the system does not revise the world model or belief state.

NoUpdate is appropriate when:

```text
evidence quality is low
signal is ambiguous
signal is empty
prediction comparison is inconclusive
source reliability is low
source is inadmissible
update target is unclear
model conflict cannot be resolved
```

NoUpdate should still record what happened.

The signal is not ignored.

It is preserved without corrupting the model.

---

## NoUpdateRecord

A `NoUpdateRecord` should explain why the model was not changed.

Possible fields:

```text
id
record_type
created_at
created_by
loop_id
status
source_refs
uncertainty
world_model_ref
evidence_ref
prediction_error_ref
blocked_update_targets
reason
required_evidence_for_update
```

Minimal fields:

```text
reason
source_refs
uncertainty
```

A NoUpdateRecord should make it clear that not updating was deliberate.

---

## NoAction

NoAction means the system does not execute or propose an external action, or it explicitly selects inaction as the safest result.

NoAction is appropriate when:

```text
evidence is weak
update did not occur
action target is unclear
permission is missing
boundary is uncertain
expected effect is unknown
outcome cannot be observed
action risk is too high
```

NoAction is distinct from NoUpdate.

The system may update the world model but still choose NoAction.

Example:

```text
The system updates project_phase to foundation_restart.
But it takes no external action because the next step is documentation in the current chat.
```

---

## NoActionRecord

A `NoActionRecord` should explain why action was not selected.

Possible fields:

```text
id

created_at
created_by
loop_id
status
source_refs
uncertainty
reason
blocked_action_names
safety_ref
evidence_ref
expected_missing_permission
required_condition_for_action
```

The first restart can keep this minimal.

---

## GatherMoreSignal

Sometimes the system should not update or act, but should request or wait for more evidence.

This is different from passive abstention.

A gather-more-signal result means:

```text
The current evidence may matter, but it is insufficient.
More observation is required.
```

Examples:

```text
ask the user to clarify
inspect approved system status
retrieve bounded memory context
rerun a safe read-only check
wait for another signal
```

GatherMoreSignal should also be bounded by safety.

The system should not gather from sensitive sources without permission.

---

## Human review

Some uncertainty should trigger human review.

Examples:

```text
prior revision requested
safety boundary conflict
sensitive sensor activation
file mutation request
ambiguous destructive command
conflicting high-confidence records
untrusted source proposes action
```

Human review should be explicit.

The system should not pretend human review happened if it did not.

---

## Evidence and human correction

Human correction can be high-quality evidence for user intention and project direction.

Example:

```text
User says:
"No, we need to start from scratch."
```

This is strong evidence about what the user wants.

But even human correction should be scoped.

It may be authoritative about:

```text
user intention
project preference
terminology preference
next task
```

It may not automatically be authoritative about:

```text
file existence
external factual claims
system state
security conditions
runtime health
```

Evidence quality should be scoped to the claim.

---

## Evidence and memory

Retrieved memory is context, not truth.

A memory trace can support evidence quality, but it should not automatically force update.

Example:

```text
Memory retrieval finds v0.2 closeout note.

Evidence use:
Supports the model that v0.2 was closed out.

Limit:
Does not prove the current repo state unless current Git status or user confirmation also supports it.
```

Memory should be source-backed and uncertainty-aware.

---

## Evidence and LLM output

If an LLM is used later, its output should be treated as a proposal signal.

It should not be treated as evidence by itself unless supported by source-backed records.

Possible LLM outputs:

```text
interpretation proposal
summary proposal
classification proposal
possible prediction
risk flag
```

These outputs should be checked against:

```text
records
signals
world model
priors
safety boundaries
```

An LLM should not be the evidence authority.

---

## Evidence and action readiness

Evidence quality should influence whether action can be proposed.

Low evidence should usually lead to:

```text
NoAction
```

Medium evidence may allow:

```text
bounded action proposal
human review
safe internal update
```

High evidence may allow:

```text
scoped update
safe action proposal
```

But even high evidence does not bypass safety.

Action must still pass safety checks.

---

## Evidence thresholds

The system may use thresholds.

Example:

```text
update_threshold = 0.6
action_threshold = 0.75
sensitive_action_threshold = human_review_required
```

Thresholds should not be arbitrary hidden magic.

They should be visible in records.

A record should show:

```text
evidence score
threshold
whether threshold was met
reason
```

The first restart may use labels instead of numeric thresholds.

That is acceptable as long as the logic is inspectable.

---

## Update threshold versus action threshold

Updating a model and taking action are different.

Action usually requires a higher bar.

Example:

```text
Evidence is sufficient to update current_project_phase.
Evidence is not sufficient to modify files.
```

The system should be able to say:

```text
allow_model_update = true
allow_action = false
```

This distinction is important.

---

## Safety threshold

Some actions should require explicit human permission regardless of evidence score.

Examples:

```text
file deletion
editing Source Documents/
running terminal commands
network access
camera activation
microphone activation
window tracking
credential access
```

Evidence quality cannot override safety boundaries.

A strong signal requesting unsafe action should still be blocked.

## Evidence in the corrected cognitive tick

The corrected tick should use evidence like this:

```text
BodyStateRecord
PriorModelRecord
WorldModelRecord
→ PredictionRecord
→ SignalRecord
→ PredictionErrorRecord
→ EvidenceRecord
→ ModelUpdateRecord or NoUpdateRecord
→ ActionProposalRecord or NoActionRecord
→ SafetyCheckRecord
→ OutcomeRecord
→ MemoryTraceRecord
```

Evidence sits between prediction error and update.

That position is important.

---

## What the first restart should prove

The first corrected prototype should prove:

```text
prediction error does not automatically update the model
evidence quality is evaluated after prediction error
low evidence produces NoUpdate
low evidence produces NoAction
clear evidence can produce scoped ModelUpdate
uncertainty is preserved in records
signals outside boundary do not update the model
```

This can be deterministic and simple.

It does not need complex scoring at first.

---

## Minimal deterministic evidence rules

The first restart can use rules such as:

```text
if signal is empty:
  evidence_quality = none
  result = NoUpdate

if signal has ambiguous_text feature:
  evidence_quality = low
  result = NoUpdate

if signal source is disabled:
  evidence_quality = inadmissible
  result = NoUpdate

if prediction_error is match and signal is clear:
  evidence_quality = medium or high
  result = eligible_for_scoped_update

if prediction_error is contradiction with safety prior:
  evidence_quality = blocked_for_action
  result = NoAction / HumanReview
```

These rules are enough for the first foundation.

---

## Tests for evidence

Recommended tests:

```text
- empty signal creates low/none evidence
- ambiguous signal creates low evidence
- disabled source creates inadmissible evidence
- matched clear signal creates medium/high evidence
- prediction mismatch does not automatically update model
- evidence reasons are recorded
- evidence references prediction error and signal
- evidence quality can recommend NoUpdate
- evidence quality can allow scoped update
```

---

## Tests for uncertainty

Recommended tests:

```text
- uncertainty can be added to records
- uncertainty reduces update readiness
- uncertainty blocks action when action target is unclear
- uncertainty is preserved in NoUpdateRecord
- uncertainty is preserved in NoActionRecord
- unresolved questions appear in memory trace
```

---

## Tests for abstention

Recommended tests:

```text
- empty signal produces abstention behavior
- ambiguous signal produces abstention behavior
- low evidence produces NoUpdate
- low evidence produces NoAction
- inadmissible signal produces NoUpdate and NoAction
- abstention includes reason
- abstention does not erase signal record
```

---

## What not to implement yet

Do not overbuild evidence in the first restart.

Avoid:

```text
complex Bayesian scoring
machine learning classifiers
LLM-based evidence judgments
embedding-based confidence
large probabilistic models
multi-source weighted fusion
hidden thresholds
```

The first version should be deterministic and inspectable.

The goal is correct architecture, not sophistication.

---

## Future evidence methods

Later versions may explore:

```text
Bayesian updating
probabilistic graphical models
source reliability calibration
multi-signal evidence fusion
anomaly detection
confidence intervals
retrieval-weighted evidence
formal contradiction detection
human feedback calibration
```

These should only be added after the corrected loop is stable.

---

## Common failure modes

### Failure mode 1: signal becomes truth

```text
Signal arrives.
System updates immediately.
```

This bypasses evidence.

### Failure mode 2: mismatch becomes proof

```text
Prediction mismatch occurs.
System assumes model is wrong.
```

Mismatch is not proof.

### Failure mode 3: uncertainty is hidden

```text
System gives a clean answer while uncertainty remains unresolved.
```

This creates false confidence.

### Failure mode 4: abstention is treated as failure

```text
System always tries to update or act.
```

This makes the system unsafe.

### Failure mode 5: evidence bypasses safety

```text
Evidence is high, so action proceeds despite missing permission.
```

Evidence quality cannot override safety boundaries.

---

## Design principles

Future work should preserve these principles:

```text
1. Signals are not evidence automatically.
2. Prediction error is not proof.
3. Evidence quality gates model update.
4. Uncertainty is first-class.
5. Abstention is intelligent.
6. NoUpdate preserves model integrity.
7. NoAction preserves safety.
8. Memory is context, not truth.
9. Human correction is strong evidence for user intention, but scoped.
10. Evidence quality cannot override safety.
11. Action requires a higher bar than model update.
12. Every evidence judgment needs reasons.
13. Every abstention needs a traceable explanation.
```

---

## Final summary

The Bounded Predictive Machine must be able to resist weak evidence.

A signal arrives.

A prediction error occurs.

But the system must still ask:

```text
Is this signal admissible?
Is it reliable?
Is it clear?
Is it strong enough to update the model?
Is it safe enough to act on?
What remains uncertain?
```

When the answer is no, the correct result is:

```text
NoUpdate
NoAction
Abstention
GatherMoreSignal
HumanReview if needed
```

This is not a failure of cognition.

It is disciplined cognition.

The corrected architecture should therefore preserve this sequence:

```text
prediction
→ signal
→ prediction error
→ evidence quality
→ uncertainty gate
→ update/no-update
→ action/no-action
```

Evidence, uncertainty, and abstention are what keep the system from becoming a reactive, overconfident, unsafe machine.
