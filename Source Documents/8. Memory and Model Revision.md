# 08 — Memory and Model Revision

## Purpose of this document

This document defines the role of **memory** and **model revision** in the corrected **Bounded Predictive Machine** architecture.

The key idea is:

```text
Memory is not storage.
Memory is reconstruction.
```

The system should not simply save summaries, user messages, or outputs.

It should preserve the chain of events that explains why the world model changed, why it did not change, why the system acted, why it abstained, and what uncertainty remained.

Memory should support future prediction, but memory should not become automatic truth.

## Core thesis

The Bounded Predictive Machine should remember by preserving traceable evidence chains.

A memory should answer:

```text
What was the prior model?
What did the system predict?
What signal arrived?
Where did the signal come from?
How did the signal compare with prediction?
What prediction error occurred?
How good was the evidence?
What uncertainty existed?
Did the model update?
If not, why not?
Was action proposed?
Was action blocked or allowed?
What outcome was observed?
What should future predictions learn from this?
```

Memory is useful only if it preserves enough structure to support future reasoning, audit, and correction.

A vague summary can be useful for humans, but it is not enough for the architecture.

---

## Memory is not storage

Ordinary storage says:

```text
save this text
save this summary
save this conversation
save this preference
```

The Bounded Predictive Machine needs something stricter:

```text
preserve the evidence chain
preserve the source records
preserve the comparison between prediction and signal
preserve why update did or did not happen
preserve uncertainty
preserve safety decisions
preserve outcome observations
```

The goal is not just recall.

The goal is reconstructability.

---

## Memory is not truth

A memory trace is evidence about what the system recorded.

It is not automatically truth about the world.

A memory trace may be:

```text
incomplete
stale
based on weak evidence
based on an ambiguous signal
superseded by later evidence
linked to a mistaken interpretation
valid for one context but not another
```

Therefore, retrieved memory should be treated as context.

It may inform prediction.

It may support evidence quality.

It may suggest model revision.

But it should not automatically overwrite the current world model.

---

## Memory and the world model are different

Memory stores traces of what happened.

The world model represents the system’s current structured understanding.

Example memory trace:

```text
At time T, the system predicted that the user might request another v0.2 implementation chunk.
The user said: "No, we need to start from scratch."
Prediction error was mismatch.
Evidence quality was high for user intention.
The model updated current_phase to foundation_restart.
```

Example world model after the trace:

```text
current_phase: foundation_restart
v0.1_v0.2_status: learning_spikes
next_priority: write handoff document pack
manual_text_role: one signal source only
```

Memory supports the world model.

Memory is not the world model itself.

---

## Memory and belief are different

A belief or world-model state is the system’s current working structure for prediction.

A memory is a record of events, evidence, and revisions.

The system should avoid this mistake:

```text
retrieved memory says X
→ current belief becomes X
```

The corrected pattern is:

```text
retrieved memory says X
→ memory is treated as contextual signal
→ compare with current model and current evidence
→ update only if justified
```

Memory can influence belief.

It should not directly become belief.

---

## Why memory matters for prediction

Future predictions should be shaped by past prediction errors.

If the system repeatedly predicts the wrong next signal, memory should help revise the world model.

Example:

```text
Repeated prediction:
The user will continue implementation prompts.

Repeated signal:
The user requests architecture documents.

Model revision:
current_phase = handoff_document_pack
expected_next_signal_features = request_for_next_numbered_document
```

Memory lets the system detect patterns over time.

But pattern detection should remain evidence-gated.

## What memory should preserve

A good memory trace should preserve references to:

```text
BodyStateRecord
PriorModelRecord
WorldModelRecord
PredictionRecord
SignalSourceRecord
SignalRecord
PredictionErrorRecord
EvidenceRecord
ModelUpdateRecord or NoUpdateRecord
ActionProposalRecord or NoActionRecord
SafetyCheckRecord
OutcomeRecord
remaining uncertainty
```

The memory trace should not flatten this chain into an unsupported summary.

It may include a readable summary, but the summary should be backed by record references.

## LoopRecord versus MemoryTraceRecord

A `LoopRecord` and `MemoryTraceRecord` are related but not identical.

The `LoopRecord` preserves the ordered structure of one tick.

It answers:

```text
Which records were created, and in what order?
```

The `MemoryTraceRecord` preserves reconstructive meaning.

It answers:

```text
What happened in this tick, why did it matter, what changed, what did not change, and what uncertainty remains?
```

Both are useful.

The loop record supports order.

The memory trace supports future interpretation and revision.

---

## Memory trace as evidence chain

A memory trace should make it possible to reconstruct the reasoning path.

Example structure:

```text
Prior world model:
current_phase = handoff_document_pack

Prediction:
next signal may request the next numbered document

Signal:
"08 memory and model revision"

Prediction error:
match

Evidence quality:
medium/high for current task continuation

Model update:
active_task = write document 08

Action:
create markdown document in canvas

Outcome:
document created

Remaining uncertainty:
repo file not yet written unless user exports or commits separately
```

This is more useful than storing only:

```text
User asked for document 08.
```

---

## Memory should preserve no-update

Memory should not only preserve successful updates.

It should also preserve cases where the system refused to update.

Example:

```text
Prediction:
next signal should contain a clear instruction

Signal:
"???"

Prediction error:
inconclusive

Evidence quality:
low

Result:
NoUpdate
NoAction

Memory:
unclear signal preserved; world model unchanged; more signal needed
```

This matters because future signals may clarify the earlier ambiguity.

A weak signal may become useful context later, but it should not corrupt the world model immediately.

---

## Memory should preserve no-action

Memory should preserve why the system did not act.

Example:

```text
Signal:
"Delete the Source Documents folder."

Prediction error:
contradiction / unsafe mismatch

Evidence quality:
high that request occurred, blocked for action

Safety:
Source Documents/ is read-only

Result:
NoAction

Memory:
unsafe request recorded; no file mutation performed; boundary preserved
```

This lets the system remember that the event happened without treating the requested action as approved.

---

## Memory should preserve uncertainty

Uncertainty should not be lost during memory writing.

A memory trace should preserve unresolved questions such as:

```text
signal meaning unclear
source reliability unknown
world model partially inferred
current repo state unverified
user intention changed but scope unclear
action outcome not observed
memory retrieval incomplete
```

Future prediction should consider these uncertainties.

A memory summary that removes uncertainty can create false confidence.

---

## Memory and source references

Memory should be source-backed.

A memory trace should reference the records that support it.

Example:

```text
MemoryTraceRecord.source_refs:
  - body-state-123
  - prior-model-456
  - world-model-789
  - prediction-111
  - signal-222
  - prediction-error-333
  - evidence-444
  - model-update-555
  - outcome-666
```

This lets the system audit why the memory exists.

---

## Memory and summaries

Readable summaries are useful.

But a summary is not enough.

Bad memory:

```text
The user wanted the project restarted.
```

Better memory:

```text
The system predicted continuation of v0.2 planning.
The user clearly corrected this by saying the project should start from scratch.
Prediction error was mismatch.
Evidence quality was high for project direction.
The world model updated current_phase to foundation_restart.
```

Best memory:

```text
Readable summary + linked source records + uncertainty + update target + outcome.
```

---

## Memory as contextual signal

Retrieved memory should enter the system as a signal-like context.

A `MemoryTraceSignalSource` or retrieval step may produce:

```text
retrieved_memory_context
matched_records
matched_fields
matched_tokens
retrieval_confidence
staleness
uncertainty
```

This should feed prediction or evidence evaluation.

It should not directly update the world model.

Correct pattern:

```text
retrieve memory
→ treat as contextual signal
→ compare with current model and current signal
→ update only if evidence supports revision
```

---

## Memory retrieval is not memory authority

v0.2 implemented local JSONL memory retrieval.

That was useful.

But retrieval must not become authority.

A retrieved memory may be relevant but wrong for the current context.

Example:

```text
Memory says v0.2 branch was active.
Current user says restart should begin from main.
Git status may show a different branch.
```

The system should reconcile these signals instead of blindly following retrieved memory.

---

## Memory staleness

Memories can become stale.

A memory trace should ideally support staleness evaluation.

Staleness may depend on:

```text
created_at
model_version
project phase
whether later updates superseded it
whether source records were later contradicted
whether the environment changed
```

A stale memory may still be useful historically.

But it should be weaker evidence for current state.

---

## Memory conflict

Memories may conflict.

Example:

```text
Memory A:
Project should continue v0.2 implementation.

Memory B:
Project should restart from corrected foundation.
```

The system should not merge these carelessly.

It should ask:

```text
Which memory is newer?
Which has stronger evidence?
Which came from direct human correction?
Which applies to current phase?
What uncertainty remains?
```

Conflict should be represented, not hidden.

---

## Memory and model revision

Model revision is the process of changing the world model, body state, or sometimes priors based on evidence.

Memory supports model revision by preserving the evidence chain behind past and current changes.

A model revision should answer:

```text
What model state changed?
What prior model state did it replace?
What evidence justified the change?
What prediction error created update pressure?
What uncertainty remains?
What future prediction should change?
```

Revision should be explicit and scoped.

---

## Types of model revision

The system may revise different kinds of model structures.

### World model revision

Most common.

Example:

```text
current_phase: v0.2_continuation → foundation_restart
```

### Body state revision

Capability or runtime state changed.

Example:

```text
SystemStatusSignalSource: unavailable → available
```

### Prior revision

Deeper structural assumption changed.

Example:

```text
memory_retrieval_weighting_policy updated after repeated retrieval errors
```

Prior revision should require stronger evidence than ordinary world model revision.

### Confidence revision

The model does not change content, but confidence changes.

Example:

```text
confidence(current_phase = handoff_document_pack): medium → high
```

### Uncertainty revision

An uncertainty is resolved, added, or changed.

Example:

```text
uncertainty: implementation branch not yet created
```

later becomes:

```text
restart branch exists and is active
```

---

## NoUpdateRecord and memory

A `NoUpdateRecord` should also feed memory.

NoUpdate means:

```text
the signal was recorded, but the model was not revised
```

A memory trace should preserve:

```text
why update was blocked
what evidence was missing
what uncertainty remains
what future signal may resolve it
```

This prevents the system from forgetting weak or ambiguous events while still protecting the model from false updates.

---

## Model revision should be scoped

A model update should change a specific target.

Bad update:

```text
Update belief.
```

Better update:

```text
Update WorldModelRecord.current_phase from documentation_handoff to restart_implementation_planning.
```

Even better:

```text
Update WorldModelRecord.current_phase based on clear user instruction signal, prediction mismatch, and high evidence quality.
Preserve uncertainty about whether the Git branch has been created.
```

Scoped updates reduce accidental overgeneralization.

---

## Model revision should preserve prior value

The system should preserve what changed from what.

Example:

```text
prior_value: current_phase = v0.2_continuation
posterior_value: current_phase = foundation_restart
```

This allows rollback, audit, and explanation.

A model that overwrites itself without preserving prior state loses its evidence history.

---

## Model revision should cite evidence

Every model revision should cite the evidence that justified it.

A model update without evidence references should be treated as weak or invalid.

Example:

```text
ModelUpdateRecord.source_refs:
  - previous-world-model-id
  - prediction-id
  - signal-id
  - prediction-error-id
  - evidence-id
```

This keeps belief change traceable.

---

## Model revision should affect future prediction

A model update matters because it changes future predictions.

Example:

```text
Update:
current_phase = handoff_document_pack

Future prediction effect:
next signal may request next numbered document
```

If a model update has no effect on future prediction, the system should ask whether it belongs in the world model or only in memory.

World model should contain prediction-relevant structure.

---

## Memory should help calibration

Memory should help the system calibrate predictions.

Examples:

```text
If predictions often mismatch, lower confidence.
If signals repeatedly match a pattern, strengthen that pattern.
If a source often produces ambiguous signals, lower source reliability.
If memory retrieval often misleads, reduce retrieval weight.
```

Calibration should happen slowly and traceably.

It should not overreact to one weak signal.

---

## Memory should support audit

A future collaborator or assistant should be able to inspect memory and ask:

```text
Why does the system believe this?
What evidence changed the model?
Which signals were ignored or treated as weak?
Which actions were blocked?
What uncertainty remains?
```

If memory cannot answer these questions, it is not sufficient for this architecture.

---

## Memory should support rollback or correction

If a model update later proves wrong, memory should help identify the source of the mistake.

Example:

```text
Wrong update:
current_phase = implementation

Later correction:
user says documentation is still the priority

Memory should show:
which signal caused the wrong update
what prediction error was used
what evidence quality was assigned
why the update was allowed
```

This allows correction without losing the learning trail.

---

## Memory and source hierarchy

Some memory traces should carry more weight than others.

Example hierarchy:

```text
Direct user correction about intention: high for intention
Current system status: high for runtime facts
Git status output: high for repo state if command was approved
Older summary memory: medium or low
LLM-generated summary: proposal only
Ambiguous text: low
```

Memory retrieval should preserve the source type so later evidence evaluation can weight it properly.

---

## Memory and privacy

Memory can preserve sensitive information.

Therefore memory should be governed.

The system should consider:

```text
what is stored
why it is stored
how long it is needed
whether it contains sensitive user data
whether it contains file paths
whether it contains credentials or secrets
whether it should be redacted
whether it should be excluded from persistence
```

Early prototypes should avoid storing secrets, credentials, broad filesystem data, private messages, camera data, microphone transcripts, or unnecessary personal information.

---

## Memory and read-only boundaries

Memory should not imply permission to mutate sources.

Example:

```text
Memory says Source Documents/ exists.
```

This does not mean the system may edit Source Documents/.

The body state and safety model still govern action.

Memory is context, not permission.

---

## Memory storage formats

Possible memory storage formats include:

```text
JSONL traces
structured JSON records
SQLite database
event-sourced log
graph database
vector index
hybrid trace + retrieval store
```

The first prototype used JSONL traces.

That was useful.

The corrected restart can begin without persistence, then reintroduce JSONL once the record order is correct.

Recommended order:

```text
correct loop records first
memory trace record second
persistence third
retrieval fourth
```

---

## JSONL memory

JSONL memory is useful because:

```text
it is simple
append-only patterns are easy
records remain inspectable
local retrieval is possible
tests are straightforward
```

Limitations:

```text
retrieval can be shallow
staleness handling is manual
conflict resolution is not automatic
large history may become hard to search
```

JSONL is a good early tool, not the final memory theory.

---

## Embedding memory

Embedding retrieval may be useful later.

But embeddings should not be added before the corrected loop is stable.

Embeddings can help retrieve semantically related traces.

They should not decide truth or update authority.

Correct future role:

```text
embedding retrieval suggests relevant memory context
```

Incorrect role:

```text
embedding similarity decides belief
```

Embeddings are retrieval tools, not cognition authority.

---

## Graph memory

A future graph memory may represent relationships among:

```text
signals
predictions
prediction errors
updates
world model states
actions
outcomes
uncertainties
source documents
project phases
```

Graph memory could help trace causality and dependency.

But this is a later stage.

The first restart should not begin with a graph database.

## Episodic and semantic memory

The system may eventually distinguish memory types.

### Episodic memory

Specific events:

```text
At this time, this signal arrived, this prediction error occurred, this update happened.
```

### Semantic memory

Generalized knowledge:

```text
The project is called Bounded Predictive Machine.
Manual text is one signal source.
Source Documents/ is read-only.
```

### Procedural memory

How to do things:

```text
How to run tests.
How to create a restart branch.
How to validate records.
```

### Safety memory

Constraints and incidents:

```text
This source is disabled.
This action was blocked.
This permission is required.
```

The first restart can avoid formal memory types, but the architecture should allow them later.

---

## Memory compression

Memory may need compression over time.

But compression can destroy uncertainty and source links.

Any memory compression should preserve:

```text
source record references
uncertainty
update decisions
no-update decisions
safety decisions
outcomes
revision history
```

A compressed summary without source links should be treated as lower-quality memory.

---

## Memory and metacognition

The system should monitor memory quality.

Questions:

```text
Is memory retrieval reliable?
Are retrieved traces stale?
Are memory summaries source-backed?
Are there conflicting memories?
Is memory coverage incomplete?
Is the current world model relying too heavily on weak memory?
```

Memory quality should affect confidence and uncertainty.

---

## Model revision and metacognition

The system should monitor its own revision behavior.

Questions:

```text
Am I updating too often?
Am I refusing updates too often?
Are updates based on strong evidence?
Are predictions improving over time?
Are repeated errors occurring?
Are action decisions aligned with safety?
```

This helps future versions calibrate update thresholds and prediction quality.

## What the first restart should prove

The first corrected restart should prove:

```text
MemoryTraceRecord links the full cognitive chain.
Memory preserves update and no-update outcomes.
Memory preserves uncertainty.
ModelUpdateRecord changes a scoped world model field.
NoUpdateRecord preserves the prior world model.
Memory does not automatically become truth.
Retrieved memory, when added later, is treated as context.
```

The first restart does not need advanced retrieval.

It only needs reconstructive memory.

---

## Minimal implementation approach

The first restart can implement memory simply:

```text
create_memory_trace(loop_records)
```

The function should produce:

```text
MemoryTraceRecord
```

with:

```text
referenced_record_ids
referenced_record_types
reconstruction_summary
remaining_uncertainty
```

Later, persistence can write these records to JSONL.

But persistence should not come before the corrected loop order.

---

## Tests for memory

Recommended tests:

```text
- memory trace references prediction, signal, prediction error, evidence, and update/no-update records
- memory trace preserves record order through LoopRecord reference
- memory trace includes remaining uncertainty
- memory trace can be created for ModelUpdate path
- memory trace can be created for NoUpdate path
- memory trace can be created for NoAction path
- memory trace summary does not replace source references
- memory trace serializes to dict and JSON
```

---

## Tests for model revision

Recommended tests:

```text
- model update changes a specific target
- model update preserves prior value and posterior value
- model update references evidence and prediction error
- no-update preserves prior world model
- weak evidence produces no-update
- clear evidence produces scoped update
- prior revision is not allowed through ordinary world model update path
- body state capability changes require explicit record
```


The first restart should keep memory local, explicit, source-backed, and minimal.

---

## Common failure modes

### Failure mode 1: memory becomes truth

```text
retrieved memory → belief update
```

This bypasses evidence.

### Failure mode 2: memory becomes summary only

```text
save vague summary without source records
```

This loses reconstructability.

### Failure mode 3: uncertainty disappears

```text
memory summary sounds confident even though evidence was weak
```

This creates false confidence.

### Failure mode 4: no-update is forgotten

```text
only successful updates are remembered
```

This loses important negative evidence.

### Failure mode 5: old memory overrules current evidence

```text
stale trace overrides newer signal
```

This corrupts the world model.

---

## Design principles

Future work should preserve these principles:

```text
1. Memory is reconstruction, not storage.
2. Memory is context, not truth.
3. Memory should preserve source references.
4. Memory should preserve prediction error.
5. Memory should preserve evidence quality.
6. Memory should preserve no-update and no-action.
7. Memory should preserve uncertainty.
8. Model revision should be scoped.
9. Model revision should preserve prior and posterior values.
10. Model revision should cite evidence.
11. Retrieved memory should be evidence-gated.
12. Memory summaries should not replace trace records.
13. Stale and conflicting memories should be represented explicitly.
14. Memory should improve future prediction without becoming authority.
```

---

## Final summary

The Bounded Predictive Machine should remember by reconstructing what happened.

A useful memory does not merely say:

```text
The user wanted X.
```

It says:

```text
The system believed A.
It predicted B.
Signal C arrived from source D.
The signal matched or mismatched prediction E.
Evidence quality was F.
The model updated or did not update for reason G.
Action was proposed, blocked, or skipped for reason H.
Outcome I was observed.
Uncertainty J remains.
```

That is memory as reconstructive trace.

Model revision should use those traces carefully.

Memory may inform prediction and support evidence.

But memory is not truth.

The corrected architecture should preserve this relationship:

```text
memory trace
→ contextual signal
→ evidence evaluation
→ scoped model revision or no-update
```

If memory becomes uncritical storage, the system may hallucinate continuity.

If memory remains source-backed reconstruction, the system can learn without losing traceability.
