# 09 — Action, Safety, and Boundaries

## Purpose of this document

This document defines the role of **action**, **safety**, and **boundaries** in the corrected **Bounded Predictive Machine** architecture.

The key idea is:

```text
Action is not obedience.
Action is a bounded intervention that must be justified, permitted, traceable, and safe.
```

The system should not act merely because a signal asks it to act.

It should only act after prediction, signal comparison, prediction error, evidence evaluation, uncertainty review, and safety checks.

When action is not justified, the system should choose:

```text
NoAction
Abstention
GatherMoreSignal
HumanReview
```

## Core thesis

The Bounded Predictive Machine should treat action as a serious runtime event.

An action should never be an automatic continuation of a user command, model output, memory retrieval, or prediction error.

The corrected action path is:

```text
prediction
→ signal
→ prediction error
→ evidence quality
→ model update or no-update
→ action proposal or no-action
→ safety check
→ outcome observation
→ memory trace
```

Action is downstream of evidence and safety.

Safety is not a decoration around action.

Safety governs whether action capability exists at all.

---

## Action is not obedience

A normal assistant-like system may treat a command as something to follow:

```text
user says do X
→ system does X
```

The Bounded Predictive Machine must not use that pattern.

The corrected pattern is:

```text
signal requests or suggests X
→ system checks prediction context
→ system evaluates evidence and uncertainty
→ system checks whether X is inside boundary
→ system checks whether X is permitted
→ system checks expected effect and observability
→ system proposes or refuses action
→ safety check allows, blocks, or requests review
```

A command is a signal.

A signal is not permission.

A signal is not truth.

A signal is not action authority.

---

## Definition of action

An **action** is a bounded intervention expected to change future signals, system state, external state, uncertainty, model quality, or evidence availability.

Examples of actions may include:

```text
create a document
edit a file
run a command
ask the user for clarification
retrieve memory
inspect approved system status
write a trace
open a tool
call an API
activate a sensor
send a message
```

Not all actions have the same risk.

Some actions are internal and low risk.

Some are external and high risk.

Some should be forbidden until explicitly admitted.

---

## Observation is not action

Observation and action must remain separate.

Observation means receiving or inspecting a signal.

Action means intervening to change something or request a change.

Example:

```text
Filesystem observation:
check whether a file exists

Filesystem action:
create, edit, move, or delete a file
```

Example:

```text
Terminal observation:
read human-pasted pytest output

Terminal action:
run pytest directly
```

Observation may still require boundaries.

But mutation requires stronger safety.

---

## Action proposal

Before action occurs, the system should create an action proposal.

An action proposal says:

```text
This is the action being considered.
This is why it is being considered.
This is what it is expected to change.
This is what evidence supports it.
This is what uncertainty remains.
This is what boundary applies.
This is what permission is required.
```

The action proposal is not execution.

It is a candidate that safety can allow, block, or send for human review.

## NoAction

NoAction is a valid and often intelligent outcome.

The system should choose NoAction when action is not justified, not safe, not clear, or not permitted.

NoAction is appropriate when:

```text
evidence is weak
signal is ambiguous
prediction comparison is inconclusive
model update did not occur
action target is unclear
expected effect is unknown
outcome cannot be observed
permission is missing
boundary is uncertain
risk is too high
source is inadmissible
human review is required
```

NoAction should be recorded explicitly.

The system should not silently fail to act.

---

Minimal fields might be:

```text

```

The record should make no-action auditable.

## Safety shell

The safety shell governs the cognitive core.

The cognitive core asks:

```text
What did I expect?
What signal arrived?
What error occurred?
How good is the evidence?
Should the model update?
Should an action be considered?
```

The safety shell asks:

```text
Is this inside boundary?
Is this source admitted?
Is this capability admitted?
Is this action allowed?
Is permission present?
Is this traceable?
Can the outcome be observed?
Can the action be reversed?
Is human review required?
Should this be blocked?
```

The safety shell does not replace cognition.

It governs what cognition may do.

---

## Safety is structural

Safety should not be added after the system becomes capable.

Safety should be part of the first architecture.

The system should represent:

```text
allowed signal sources
blocked signal sources
allowed actions
blocked actions
read-only paths
writable paths, if any
permission-required capabilities
human-review-required capabilities
reversible actions
irreversible actions
trace requirements
recovery requirements
```

A capability should not become available merely because code can perform it.

It must be admitted.

---

## Capability admission

Capability admission is the process by which a signal source or action type becomes available to the system.

A capability should have:

```text
name
type
scope
permission status
risk level
allowed operations
blocked operations
trace requirements
revocation method
human review requirements
```

Examples:

```text
ManualTextSignalSource:
allowed, low risk, explicit user-provided input only

SystemStatusSignalSource:
allowed, read-only, minimal system facts only

CameraSignalSource:
disabled until explicit permission and visible activation exist

FileWriteAction:
blocked until scoped write policy and approval exist

TerminalCommandAction:
blocked until approved command policy exists
```

Capability admission is central to safety.

---

## Boundary

A boundary defines what the system may observe, modify, request, or affect.

Boundaries may apply to:

```text
files
directories
terminal commands
network access
sensors
memory stores
external APIs
user data
system settings
runtime configuration
```

A boundary should be explicit, not implied.

The system should know:

```text
inside boundary
outside boundary
boundary unknown
permission required
permission denied
human review required
```

---

## Read-only boundaries

Some areas may be observable but not mutable.

For this project, important read-only defaults include:

```text
Source Documents/ should remain read-only unless explicitly changed.
Historical prototype tags should not be rewritten casually.
Generated trace files should not be committed unless intentionally selected.
Sensitive system areas should not be modified.
```

Read-only means:

```text
the system may refer to or inspect if allowed
but may not edit, delete, move, overwrite, or rewrite
```

Read-only should be enforced by action safety.

---

## Source Documents boundary

`Source Documents/` is a special boundary.

The project history indicates that source documents should remain read-only unless the user explicitly changes that boundary.

The system may use source documents as context if provided or explicitly allowed.

But it should not modify them by default.

Forbidden by default:

```text
edit Source Documents/
delete Source Documents/
rename Source Documents/
move Source Documents/
rewrite historical source material
summarize as replacement for source without preserving original
```

Allowed only if explicitly approved:

```text
copy selected source content into a derived document
create new documentation outside Source Documents/
quote or cite source material in a new document
```

---

## Filesystem safety

Filesystem actions are risky because they can destroy or corrupt work.

Early restart defaults should be:

```text
no autonomous file mutation
no deletion
no moving files
no overwriting files
no editing Source Documents/
no broad filesystem scans
project-root observation only if admitted
```

Later, file actions may be introduced under a policy such as:

```text
write only to generated docs folder
never overwrite without confirmation
show diff before commit
keep source docs read-only
record action and outcome
```

---

## Terminal safety

Terminal access must be treated carefully.

Terminal commands can:

```text
modify files
install dependencies
access network
reveal secrets
change system settings
run destructive operations
commit or push code
```

Early defaults:

```text
no autonomous terminal actions
human-pasted terminal output may be treated as signal
approved read-only commands may come later
mutating commands require explicit permission
```

Even read-only commands should be scoped.

Example safe candidates later:

```text
git status
pytest
python --version
```

Riskier commands require more caution.

---

## Git safety

Git actions can affect project history and remote state.

Early default:

```text
system may suggest Git commands
system should not execute Git commands autonomously
commits require explicit user action or approval
pushes require explicit user action or approval
history rewriting requires special caution
```

Actions requiring explicit approval:

```text
git commit
git push
git reset --hard
git clean
git rebase
git tag deletion
force push
```

The system should preserve save points and avoid destructive Git operations.

---

## Network safety

Network access can leak data, fetch untrusted content, or create external effects.

Early default:

```text
no autonomous network access
no web calls from runtime
no external API calls
no package installs without approval
```

Network access should be admitted only with clear purpose and scope.

Examples of later admitted network use might include:

```text
fetch public documentation
query a specific approved API
push to GitHub after user approval
```

But each requires boundary and permission.

---

## Sensor safety

Sensitive sensors require explicit permission and visible control.

High-sensitivity sources include:

```text
camera
microphone
active window state
keyboard/mouse activity
screen contents
browser contents
private messages
```

Early defaults:

```text
camera disabled
microphone disabled
window state disabled
keyboard/mouse tracking disabled
screen capture disabled
```

Before enabling any sensitive sensor, the system must have:

```text
explicit user permission
clear activation state
purpose limitation
bounded capture window
trace of use
privacy constraints
revocation method
human override
```

---

## Memory safety

Memory can store sensitive or misleading information.

Memory safety requires:

```text
avoid storing secrets
avoid storing unnecessary personal data
preserve uncertainty
preserve source refs
mark stale memories
avoid treating memory as truth
avoid unbounded memory ingestion
```

Generated memory should be trace-backed.

Memory writes should not silently convert uncertain signals into confident claims.

---

## LLM safety

If an LLM is used later, it should not become the authority for action.

LLM outputs may be:

```text
interpretation proposals
summary proposals
prediction proposals
risk flags
language renderings
```

LLM outputs should not be:

```text
belief authority
action authorization
safety override
capability admission authority
memory truth authority
prior revision authority
```

An LLM output is a signal or proposal.

It must be bounded, checked, and traced.

---

## Human approval

Some actions require human approval regardless of evidence score.

Examples:

```text
modifying files
committing code
pushing to remote
running terminal commands
installing dependencies
changing safety boundaries
enabling camera or microphone
enabling window state observation
using network access
changing priors
editing Source Documents/
```

Human approval should be explicit.

The system should not infer approval from vague signals.

---

## Human approval is scoped

Approval should be scoped to a specific action.

Bad approval interpretation:

```text
User approved one file write.
System assumes all future writes are allowed.
```

Correct approval interpretation:

```text
User approved writing this document to this path for this purpose.
Future writes still require policy or approval.
```

Approval should specify:

```text
action
target
scope
timeframe
risk
expected effect
```

## Safety outcomes

A safety check may produce:

```text
allowed
blocked
requires_human_review
requires_more_information
outside_boundary
permission_missing
capability_not_admitted
```

The system should not collapse all safety failures into generic refusal.

Different safety outcomes teach different lessons.

## OutcomeRecord

An `OutcomeRecord` records what happened after action or no-action.

Possible fields:

```text
id
record_type
created_at
created_by
loop_id
status
source_refs
uncertainty
action_ref
no_action_ref
safety_ref
expected_effect

failure_reason
unresolved_questions
outcome_observed
```

If no external action occurred, the outcome should say so.

Example:

```text
observed_effect: no external action performed
success: true
```

This prevents confusion between proposed and executed actions.

---

## Expected effect

Every action proposal should include an expected effect.

If the system cannot say what an action is expected to change, the action is not ready.

Examples:

```text
Action:
create document 09 in canvas

Expected effect:
a markdown document titled 09-action-safety-boundaries exists in the canvas
```

```text
Action:
run pytest

Expected effect:
test output signal reports pass/fail state
```

```text
Action:
ask user for clarification

Expected effect:
future manual text signal resolves uncertainty
```

---

## Outcome observability

The system should prefer actions whose outcomes can be observed.

Questions:

```text
Can the system observe whether the action happened?
Can the system observe whether it succeeded?
Can the system observe unintended side effects?
Can the system record the result?
```

If an outcome cannot be observed, action risk increases.

Unobservable action should often produce NoAction or HumanReview.

---

## Reversibility

Action safety should consider reversibility.

Low-risk reversible actions:

```text
create a draft document
produce a proposed command
write a temporary trace
ask a clarification question
```

Higher-risk actions:

```text
edit important files
delete files
push to remote
change system settings
install packages
enable sensors
send external messages
```

Irreversible or hard-to-reverse actions require stronger approval.

---

## Action risk levels

The system may classify action risk.

Example labels:

```text
none
low
medium
high
critical
```

Examples:

```text
NoAction: none
Ask clarification: low
Create new generated doc: low/medium
Edit existing project file: medium
Run read-only test command: medium
Install dependency: high
Delete file: high/critical
Enable microphone: high
Force push Git history: critical
```

Risk should affect safety and approval requirements.

---

## Internal action versus external action

The system should distinguish internal and external actions.

Internal actions:

```text
create record
write memory trace
update internal world model
mark uncertainty
select no-action
```

External actions:

```text
write file
run command
call network API
activate sensor
send message
change system setting
commit to Git
```

Internal actions still need traceability.

External actions require stronger safety.

---

## Model update is not external action

A model update changes the internal world model.

It is not the same as external action.

However, model updates still require evidence.

Example:

```text
Update current_phase to handoff_document_pack
```

This is internal.

But it should still cite evidence.

External action would be:

```text
write handoff document to repository
```

That requires additional safety and permission.

---

## Asking the user as action

Asking the user for clarification can be an action.

It is usually low risk, but it still has a purpose.

Expected effect:

```text
future manual text signal reduces uncertainty
```

The system should ask only when it genuinely needs clarification.

It should not ask unnecessary questions when the next safe step is clear.

---

## Gather-more-signal as action

Gathering more signal can be an action if it involves actively requesting or inspecting something.

Examples:

```text
ask user for clarification
run approved system status check
retrieve bounded memory context
inspect project directory if allowed
```

Gathering more signal must respect source boundaries.

The system should not gather from camera, microphone, window state, broad filesystem, or network without permission.

---

## Safety and prediction

Safety should influence prediction.

The system should not predict actions or signals from disabled capabilities.

Example:

```text
CameraSignalSource is disabled.
```

Therefore:

```text
The system should not predict visual observations from camera.
```

Example:

```text
FileWriteAction is not admitted.
```

Therefore:

```text
The system should not predict that it will write files autonomously.
```

Safety grounds the world model.

---

## Safety and evidence

Evidence quality cannot override safety.

Even if evidence is strong, action may still be blocked.

Example:

```text
Signal clearly asks to delete a folder.
Evidence quality is high that the request was made.
Safety blocks deletion because permission and boundary are missing.
```

The system can update that the request occurred.

It cannot execute the unsafe action.

---

## Safety and memory

Safety decisions should be preserved in memory.

A memory trace should record:

```text
what action was proposed
why it was proposed
what safety checked
whether it was allowed or blocked
what outcome occurred
what uncertainty remains
```

This helps future auditing and calibration.

---

## Safety and capability expansion

Capability expansion should be staged.

A safe ladder might be:

```text
Stage 1: internal records only
Stage 2: manual text signal
Stage 3: system status signal
Stage 4: read-only project observations
Stage 5: memory retrieval
Stage 6: approved read-only terminal observations
Stage 7: proposed file writes with human approval
Stage 8: limited safe file writes
Stage 9: sensitive sensors only with explicit permission
```

The system should not jump from manual text directly to broad autonomy.

## What the first restart should prove

The first corrected restart should prove:

```text
action is downstream of evidence
NoAction is valid
SafetyCheckRecord can allow or block proposed action
low evidence produces NoAction
unsafe boundary produces NoAction or blocked safety result
OutcomeRecord distinguishes proposed action from executed action
memory trace preserves safety decision
```

It does not need to perform real external actions.

At first, it can only create action proposals and no-action records.

---

## Minimal implementation approach

For the first corrected restart, implement only:

```text
ActionProposalRecord
NoActionRecord
SafetyCheckRecord
OutcomeRecord
```

Use deterministic rules.

Example rules:

```text
if evidence_quality is low:
  create NoActionRecord

if action target is outside boundary:
  safety = blocked

if action requires unadmitted capability:
  safety = blocked

if action is internal and low risk:
  safety = allowed

if action is external or mutating:
  safety = requires_human_review
```

No real file, terminal, network, or sensor actions are required for the first restart.

---

## Tests for action

Recommended tests:

```text
- action proposal includes expected effect
- action proposal references evidence or model update
- low evidence creates NoAction
- ambiguous signal creates NoAction
- NoAction includes reason
- action is not created before evidence evaluation
- action is not executed without safety check
```

---

## Tests for safety

Recommended tests:

```text
- safe internal action can be allowed
- external mutation requires human review
- Source Documents/ mutation is blocked
- disabled capability is blocked
- missing permission blocks action
- safety check references action proposal
- safety check records blocked reason
```

---

## Tests for outcome

Recommended tests:

```text
- allowed action creates outcome record
- blocked action creates no-external-action outcome
- NoAction creates outcome record saying no action performed
- outcome references safety and action/no-action records
- outcome preserves unresolved questions
```

## Common failure modes

### Failure mode 1: command becomes action

```text
User says do X.
System does X without evidence or safety.
```

This violates the architecture.

### Failure mode 2: evidence bypasses safety

```text
Evidence is strong, so system acts despite missing permission.
```

Evidence does not override boundaries.

### Failure mode 3: proposal is confused with execution

```text
System proposes an action and records it as completed.
```

Outcome must distinguish proposed, allowed, executed, blocked, and observed.

### Failure mode 4: no-action is treated as failure

```text
System always tries to do something.
```

NoAction is often the correct result.

### Failure mode 5: capabilities silently expand

```text
A new source or action becomes available without admission record.
```

This breaks safety.

---

## Design principles

Future work should preserve these principles:

```text
1. Action is not obedience.
2. A command is a signal, not permission.
3. Observation and action are separate.
4. Action must have an expected effect.
5. Action must be observable where possible.
6. NoAction is valid and intelligent.
7. Safety governs capability.
8. Evidence quality cannot override safety.
9. Sensitive capabilities require explicit permission.
10. External actions require stronger safety than internal updates.
11. Source Documents/ remains read-only unless explicitly changed.
12. Terminal, filesystem, network, camera, microphone, and window access require staged admission.
13. Human approval is scoped, not global.
14. Outcomes must distinguish proposal from execution.
15. Safety decisions must be preserved in memory.
```

---

## Final summary

The Bounded Predictive Machine should be able to act, but only after it has earned the right to consider action.

The action path is:

```text
prediction
→ signal
→ prediction error
→ evidence quality
→ update/no-update
→ action proposal/no-action
→ safety check
→ outcome
→ memory trace
```

A safe system must be able to say:

```text
I will not update.
I will not act.
I need more evidence.
This is outside boundary.
This requires human review.
This capability is not admitted.
```

Action is a bounded, safety-governed intervention.

It is not obedience.

It is not automatic tool use.

It is not proof of intelligence.

The foundation must make safe inaction just as explicit as safe action.
